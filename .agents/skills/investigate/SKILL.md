# investigate

复刻你反复手打的调查模板(history ×5:「只读分析不要改任何代码。背景…任务:读 <file>…给方案…最后写结论报告。开始调查」)。核心纪律:**只读、挖真根因、落一份报告**。

## 铁律

- **全程零代码改动。** git diff 必须始终干净;想动手时停下,把修法写进报告让用户拍板(你反复要求「先记录不要现在就改」「先看不要改」)。
- **挖到能解释全部现象的那一个根因才停。** 症状反复 = 根因没找到;别停在「加个兜底/gapfill」这类症状层(记忆 working-style-evidence-first、footprint 惨痛反例)。
- **别猜。** 查不到就找参考实现(如「找 cc-switch 源码直接抄」)/vendor 官方文档/web 搜索,用事实说话。

## 执行步骤

1. **对齐输入**:背景是什么现象?要读哪些文件/日志?有没有红线(已知负收益方向,如编译链的族摊销/骨架预检索——见 `redline-reviewer` agent)?缺了先问清,别自己脑补范围。

2. **fan-out 读**:面广时派 Explore / general-purpose agent 并行读多个子系统,只收结论不搬文件原文。设备侧现象拉真实日志/配置对照(照妖镜是时间戳因果)。

3. **定位根因**:给铁证——`file:line` 引用、日志原文子串、先例对照。分清「共性/系统性」vs「个案」。核对是不是本次改动引入的(git diff / stash 对照)。

4. **写结论报告**落 `docs/`,命名随现有惯例(`DIAG_*` 诊断 / `RESEARCH_*` 调研 / `AUDIT_*` 审计 / `REVIEW_*` 评审)。结构:现象 → 根因(带证据) → 影响面 → 修法方向(不落地) → 红线/遗留。

## 注意

- 报告要能被机读断言核对(把要防的回归写成「产出不含 X」这类可验证句),别只靠肉眼看一次(AGENTS.md eval-first)。
- 结论给数据契约不给模糊描述;修法方向标清哪些「应该做/不应该做」。
- 若这是编译链/prompt 改动的评审,直接调 `redline-reviewer` agent 而不是自己临时判红线。
