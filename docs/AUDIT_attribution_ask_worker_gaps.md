# 彻查报告:归因→问询→编译三链的架构污点(2026-07-13 run19 实弹驱动)

> 状态:只读彻查产出,零代码改动。**术语语境:全文为软件测试自动化领域标准术语**——「测试床
> 状态污染」= 一个测试用例的配置残留影响后续用例(testbed state contamination)、「归因」= 失败
> 根因分析(failure attribution)、「s₀」= 初始状态假设、「污染源/受害者」= 用例间的残留传播关系;
> 纯编译引擎的归因逻辑与生成-验证架构分析,不涉及任何其他领域。
>
> 触发=run19@79 全新跑 668030/668044 被 s₀ 诊断判「测试床状态污染,建议降级」,人工逐层核对完整
> 设备回显后证伪——真根因是这两案自己的命令写法(write all/net 撞交互 YES 提示 + config all
> 子网冲突),失败机理是**自身命令流错位**,与前驱残留无关。本报告不提「补某条命令」或「加 if-then
> 规则」——那是症状层。挖的是三个结构性污点的理论/设计根。

## 0. 一句话共同根

三污点同一个理论根:**验证盲区定理说「上机完整回显是唯一语义权威」,但引擎在归因侧(diagnose
用 IP 签名+卷面画像,不读手边的 device_context)和编译侧(worker 提交前结构性看不到自己序列的
完整回显)两处都用了更弱的代理(签名/画像/先例模仿)代替权威源**。问询侧再把归因侧的错误结论零
校验透传给用户。**权威事实源在手边或本可获得,系统却用代理下判断**——这正是人工调查时我反复犯、
被用户三次纠正的同一个错,它固化在了引擎结构里。

## 1. 污点一:s₀ 归因误判——有完整回显不看,用签名代理代替失败机理

### 1.1 实证(run19@79,铁证在设备完整会话)

668030 卷面 `write all file sdns_file_save_030` → 撞上一轮残留同名文件 → 设备交互问
`Type "YES" to overwrite` → worker 无 YES 处理步 → 下一条 `no sdns listener` 被当成 YES 的
回答吃掉 → `Write aborted / Failed to execute` → 命令流错位 → 后续断言窗口全乱 → 首个 check
(found 172.16.34.70)读错窗口 fail(尽管 `show sdns listener` 明明显示 172.16.34.70 已配上)。
`config all file` 又加载全部配置 → `Error: this subnet is already used by port1/2/3`。

**同族对照(同床同轮):** 668000(`write memory`)、668015(`write file`)写法正确,**都 pass**;
668030(`write all file`)、668044(`write net`)写法错,**都 fail**。s₀ 诊断判 668030「被
668000/668015 污染」。

> **重要更正(理论对照后):** 我最初据此论证「污染源自己 pass → 排除污染假设」。理论对照证明这是
> **错误方向**——s₀ 模型是「前驱写-存活」独立于前驱 verdict(S 代数 h-in-s₀),「跨面洗白路径」构造
> 明确就是一个自己 pass 却影响下游残留的前驱(`THEORY_target_system_algebra.md:104-106`),且 668000
> 假阳已作为「接受的假阳」在册(`:162`)。按「污染源必须 fail」改判据会与理论冲突、制造新误判。
> **真正的污点不是「没验污染源 pass/fail」,而是「没读受害者完整回显确立失败机理」**(见 §1.3)。

### 1.2 代码根源(`nodes.py::_s0_pair` / `diagnose`)

- `_s0_pair`(nodes.py:1356)遍历整卷组成 `comp`,**只要前驱有持久面写(`p["persist"]`)或共享
  L2/L3 实体就列为污染者**——判据入参是卷面画像(`_case_touch_profile`,只读 xlsx 命令文本)+
  IP 签名(`sig`),**从不读 device_context**。
- `diagnose`(nodes.py:1458)`for aid in failed` 只取 `verdict.signatures`(IP 正则)喂 `_s0_pair`;
  唯一反驳 `_cross_bed_refuted`(nodes.py:1469)只管「跨床同签名」,**不管污染源自身 verdict**。
- G6 前筛(nodes.py:1014)读了 last_run(`recs_pre`,device_context 就在里面)却只用 signatures;
  且 s₀ 命中 → **跳过深归因 fork**——而 attributor fork 是唯一读 device_context 原文判层的节点。
  **两者叠加:s₀ 案的失败机理没有任何节点分析**(G6 有回显不用,又跳过了会用它的 fork)。

### 1.3 核心判据污点:用签名代替机理,失败机理从不经 echo 确立

- **污点(结构性):s₀ 判定用签名+卷面画像代替完整回显机理。** s₀ 判定全程只有 IP 正则签名 +
  卷面静态画像(`_case_touch_profile` 读 xlsx 命令文本),**受害者为什么 fail 的真实机理(完整
  设备回显)从不进入判定**。668030 的 fail 机理是「自身命令流错位(unhandled Type YES → 吃掉
  下条命令 → 断言窗口错位)」,写在 device_context 第一屏,但 diagnose/`_s0_pair` 结构上不读它。
- **后果:一个「自身 broken/intrinsic」的 fail 被强行塞进「共享实体的 s₀ 污染」桶。** 因为 s₀
  的共享实体判据「按构造必然命中」——只要受害者和某个有持久面写的前驱碰了同一实体(同族测试必然
  碰同一 IP:port),就判污染。真实失败在受害者自己那里(命令写法),回显里清清楚楚,判据却不看。
- **这不是加「验污染源 pass/fail」规则(那条被理论反驳,见 §1.1 更正)**,而是**把 s₀ 归因的
  信息层从「签名代理」升级到「完整回显机理」**——即让机械归因也遵守验证盲区定理(上机回显是唯一
  语义权威)。当前只有 LLM attributor 被强制「evidence 必须是 device_context 逐字子串」的 echo 门
  (`fail_attribution.py:189`),而真正 sentence h_s0 并驱动面板的机械 adjudicator(`_s0_pair`)
  **豁免任何 echo 门**——权威原则被倒置到了较弱的 actor 上(见 §4 映射 1B)。

## 2. 污点二:生成-验证隔离——worker 提交前看不到自己序列的完整上机回显

### 2.1 结构证据(子agent 完整取证,CONFIRMED)

- worker 工具白名单(`compile-worker.md:4`)**无任何上机执行通道**(无 dev_run_case/dev_run_batch);
  唯一活设备通道 `dev_probe` 是单条只读 show/get(`run_case.py:413-418`),跑不了 mutating 命令、
  跑不了序列、看不到运行时交互提示。
- 首轮编写完全盲写(`briefs.py:67` `is_retry` 首轮为假,brief 只有信封+意图,零设备证据)。
- 完整回显(device_context)只在**上机 fail 后**捕获(`run_case.py:149`),且只在**重编轮**反应式回灌
  (`briefs.py:79-85`,仅最新失败轮内联、6000 字符上限)——是**上一版卷**的回显,永远不是**本轮正在
  写的序列**的预览。
- `compile_precedent` 返回的是先例卷的**书面命令链**(xlsx 的 E/F/G 列),**不是执行它的设备回显**,
  且不含交互提示处理(`precedent_tools.py:97-98,358-362`)。

### 2.2 为什么「有时对有时错」是这个设计的必然

668030 的失败模式(交互 YES 吃命令→序列错位、config all→子网冲突)是**运行时、有状态、序列级
的属性,只有在设备上执行完整序列才可观测**。worker 编写时结构上从不执行完整序列,所以:
- precedent 模仿碰巧复现出能跑的形态 → pass(668000/668015);
- 碰不上 → fail(668030/668044)。
**判别信号(这条序列的完整回显)在 worker 提交前根本不存在于它的上下文里**——这就是「不是补命令
或 if-then 能解决的」的精确含义:worker 侧根本没有那个能触发判断的信号。

## 3. 污点三:归因错 → 问询错(错误判定零校验透传给用户)

### 3.1 传导链

`_s0_pair` 的错误输出(basis + polluters)→ diagnosis 事实 → ask_contradiction 的 bed 分支把
`diagnosis.basis + polluters` 直接填进 `item["evidence"]`(nodes.py 消化循环 bed 分支)→ 渲染成
用户面板题面「疑似测试床污染,polluter(s):668000、668015,建议:挂起/床已处理/如实降级」。
**问询题面 = 归因结论的直接投影,中间零校验**。

### 3.2 后果(实弹)

用户拿到的是一道**建立在错误归因上的判断题**:三个选项(挂起/床已处理复跑/如实降级)**全部预设
了「这是床污染」**,没有「这案自己写错了,该重编」的出口。理性用户按题面只能选降级——一个**本可
重编修好的案被错误降级**;或付出高昂成本(像本次人工逐层挖完整回显)才能发现题面本身是错的。

## 4. 理论/设计映射(每条给出理论论断原文+定位;判定=抵触/未覆盖/理论对实现错)

一个贯穿全篇的实现事实先摆出来(锚定三节判定):**理论有对的北极星「上机回显是唯一语义权威」
(`THEORY_k_state_machine.md:112`),且在 LLM proposer 上强制了**——`submit_attribution` 机械门要求
evidence 必须是该案 device_context 的**逐字子串**(`fail_attribution.py:189-190,238-239`),且
attributor 的 h_position 只是**候选**「批级 diagnose 才 adjudicate,你只提议不判决」(`:212-213`)。
**但真正 sentence h_s0 并驱动面板的机械 adjudicator(`_s0_pair`/`diagnose`)没有任何 echo 门**——
权威原则被强制在了较弱的 actor 上,豁免了较强的那个。

### 污点一映射

- **1A 必要非充分判据(理论对实现错):** S 代数明说共享分量是**必要非充分**——
  `THEORY_target_system_algebra.md:60`「⇐ 方向为必要非充分…充分化需补『受害方观测 π 依赖该分量
  所在面』」,`:102`「判据是必要非充分的上界…命中而不发作的情形预期存在」,`DESIGN_v8_engine.md:660`
  (S10)充分化=「加『B 的观测 π 依赖该分量所在面』细化降假阳」。**实现只落了必要半(`_s0_pair`),
  充分化细化整条省略**——把必要条件当成了充分条件用。
- **1B echo 权威倒置(理论对实现错):** 验证盲区定理「上机是唯一语义权威」(`k_state:112`)在 LLM
  proposer 上有 echo 门(`fail_attribution.py:189`),而 sentence h_s0 的机械 adjudicator 无 echo 门。
  **权威原则倒置到较弱 actor**。
- **1C 失败分类学不完整(未覆盖):** B1 三档「s₀/π/真环境故障**单案观测等价**」(`algebra:131`)
  **漏了「自身 exec/broken 失败」这一类**——受害者自身命令流错位是**单案 echo 可判**的,不需要批级。
  理论后来用 (43)(44) broken 态点补(`k_state:665-670`)、§18.5 setup 相位扫描、§18.8 可达性前筛
  (`DESIGN:995-1010`),但每个都是**窄的 marker 闭集点补**,不是通则「s₀ 归因前必须先用完整回显
  确立受害者失败机理」。run14(`DESIGN:995-1010`)是**同型**——「s₀ 交换子配对(必要条件,失联批里
  共享实体恒命中)批量误诊为床污染,11 题全部乱断言」,证明「共享实体恒命中而真因在别处」这个模式是已知
  的、只被反复点补。

### 污点二映射

- **2A 生成/验证隔离已设计,自 echo 验证未理论化(未覆盖):** 设计认可隔离——`DESIGN_v8_engine.md:1171`
  「worker 只生成不自评(生成/评估隔离);语义终判在上机 oracle」,工具边界 `compile-worker.md:3-4`
  「Generation only; never runs on-device」。**但理论从未把「生成器提交前看到自己完整序列的上机回显」
  当作 required 或 forbidden——这是缺的一环**。`dev_probe` 只给单命令语法 echo(`DESIGN:1168`),从不是
  整序列语义预演。
- **2B 为什么这个闭环被扣住(未覆盖,理论内张力):** 写法不一致正是「规划质量 ≤ 模型质量」症状
  (`algebra:142`,理论说预期内);而 observe-then-assert 禁令(`DESIGN:1123`,照抄设备回显=假验证)
  正是「naive 生成→读自己回显」危险的**原因**。**理论没解决这个张力**——没规定一种**安全的**自 echo
  一致性检查(回显只用于内部一致性判断,不抄进断言)。

### 污点三映射

- **3A (41) 管答案保真不管问题正确(未覆盖):** (41) 每一子款管的都是**答案**的往返
  (`k_state:616-628` 传输/对位/消化/提交保真,`:613` 面板→人→答案→引擎消化)。**(41) 对「呈给用户的
  问题是否建立在可靠归因上」零不变量**——错误归因和正确归因被同样忠实地传输。直接印证「错误归因
  → 忠实传输的错误问题」。
- **3B (35) 三链完备性审计跳过了归因这一跳(未覆盖):** (35) 逐跳列表
  (`k_state:449-452`「作者意图→脑图→卷面→执行→回显→digest→台账→报告→用户,每跳必须有(不变量、
  检测器、修复路)三件套」)**没有 echo→diagnosis/attribution→panel 这一跳**——(35) 要求每跳都有的
  三件套,恰恰在 s₀ 结论产生处缺席,(35) 自己的完备性被内部违背。
- **3C 唯一现存缓解只校准措辞不校准正确(未覆盖,部分缓解):** §18.8「题面证据强度校准」
  (`DESIGN:1008-1010`)识别了过度断言(假阳 20-26%),但**只软化措辞**(「判为污染」改「疑似…非
  确证」),没建立归因正确性——**软化措辞的错误归因,仍把用户导向降级**。

**一句话综合:** 理论有对的北极星且在 LLM proposer 上强制了,但(a)豁免了真正 sentence s₀ 的机械
adjudicator(污点一,理论对实现错);(b)从未理论化一种安全的生成器自 echo 一致性检查(污点二,
未覆盖);(c)用 (41) 管答案保真却不管问题的归因正确(污点三,未覆盖)。三者的公因子=**在归因和
编译两处,最权威的事实源(完整回显)在手边或本可获得,系统却用更弱的代理(签名/画像/先例模仿)
下判断**。

## 5. 为什么这不是「补命令/加规则」能解决的(架构判断)

- 污点一:让机械归因也读 device_context 确立**失败机理**(而非只看签名代理)——不是加 if-then,是
  **把 s₀ 归因的信息层从「签名+画像代理」升级到「完整回显机理」**,即让机械 adjudicator 也遵守验证
  盲区定理(上机回显是唯一语义权威)。**不是**「排除自身 pass 的污染源」(那条被 S 代数反驳)。
- 污点二:worker 侧无论加多少命令规则都无效,因为**判别信号在提交前不存在**;结构解只有两类方向——
  ①把「完整序列的一次预演回显」引入编写闭环(让 worker 提交前看到自己序列的回显),或
  ②承认编写盲写、把「首版必上机取回显再定稿」作为流程(生成-验证不再隔离)。二者都是**流程结构
  改动**,非规则补丁。
- 污点三:(41) 问询链保真管的是「答案从呈现到消化的传输/对位/提交」,**不管「问题本身是否建立在
  可靠归因上」**——这是问询链上游的未覆盖环:错误归因 → 忠实传输的错误问题。需要一条「问询前提
  校验」不变量:呈报给用户前,归因结论必须**能被受害者完整回显机理支持**(如判 s₀ 前先确认失败机理
  不是自身 broken/intrinsic——用回显判,不是用签名判)。
