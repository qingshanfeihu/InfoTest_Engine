## 根因分析（设计与理论层，2026-07-10 补）

**精确机制链**：`attribute` 节点工作清单取自状态机（`in_state(FAILED_ACTIVE)`）而非 oracle 裁决——终验后该集合为空→`nothing_to_do`，digest 的 3 个 fail 从未被读。子集轮的 oracle↔状态机对齐是隐式的（跑的恰好=failed_active 集）；终验时「上机集≠failed_active 集」，隐式对齐断裂。

**根因四层**：
- **R1 理论**：合取③量词被静默收窄（交付要求整卷语境 ∀-pass，ledger 存的是无语境 ∃-pass）。case 状态机没有得到 K 状态机的语境锚纪律（validity+observed_under+anchor）——同一体系内，知识面对矛盾观察「共存+仲裁」，case 面对矛盾观察「销毁」。
- **R2 机制**：pass 锁按迁移边设防（防弱权威改写），未按证据权威设防——同一 oracle 带更强语境无 override 通道（frozen 有配对通道，passed 没有；违背「门的条件常与远处机制成对」教训）。δ(passed, oracle_fail) 是未定义迁移，实现以恒等补全=最危险补全。
- **R3 假设**：「per-case 清场⇒子集≡整卷」被当精确律；终验作为这条假设的审计器，权威却低于被审计假设。保存族恰在假设声明边界外（清场只清运行配置，write 产物跨案存活）造出反例。
- **R4 过程**：终验路由修复的 eval 断言「终验发生」而非「裁决被消费」；信号闭集的对抗性补全只做了 K 状态机、未做 case 状态机——「oracle 反证终态」事件类从未被枚举。

**为什么没触发 ask_user**：引擎仅两条 ask 边（编写期欠定 interrupt / 终局 escalated 转述），皆在合法状态迁移下游；「oracle 反证已锁终态」进不了状态机，故到不了任何下游。问询纪律不对称：有「先问后落」（写前不确定必问），无「矛盾即问」（锁后反证必问）——而「交付 23 真 vs 26 名义」按如实报告红线恰是必须交用户的裁决。

**统一病理（#7+#10+瞬态护栏 dead code 三案同根）**：digest→ledger 无「必达」契约，护栏产物皆「可被读」而非「必被消费」，消费端按自有键（路径/状态）选择性读取。缺失公理——**oracle 残差公理**：每条上机裁决必须 ①改变状态 或 ②显式核对一致 或 ③产生显式残差（信号/升级/问询），禁止无痕吸收；即 digest→ledger 是按 autoid 的全射对账。
