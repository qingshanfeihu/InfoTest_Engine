# Claude Team 工作流（cmux claude-teams）

> 2026-07-17 实战定型：六角色团队在一天内完成四脑图批次编译交付 + 全仓审计修复轮，
> 期间经历 15+ 次 API 瞬断、一次本机断网中断、一次 P0 宪法击穿缺陷、一次成员伪造上报——
> 本文档的每条纪律都对应其中一个真实事件（见各条【实证】注）。适用于任何多 agent 协作任务。

## 1. 组建

- 入口：`cmux claude-teams` 环境下，一条消息内多个 Agent 调用 spawn named teammates（每人独立 split pane）。
- **六角色模板**（按任务裁剪，名字即职责）：

| 角色 | 域 | 职责要点 |
|------|-----|---------|
| Theory | docs/THEORY_* | 理论文档审计/修复；工程改动的**理论一致性评审**（不变量恢复、公理相容） |
| Design | docs/DESIGN_*、CLAUDE.md、docs 整编 | 设计文档审计/修复；工程改动的**设计一致性评审**（条款同步、消费面影响、体验取舍） |
| LLM-Eng | skills/、agents/、prompt、观测 | skill 规范审计/修复；langfuse 观测；ask 质量语义分析 |
| Py-Eng | main/、tests/ | 代码审计/修复；机械事实取证（mirror 解析、xlsx 解析、行级根因） |
| TUI-Eng | ink/、tui/ | 渲染审计/修复；事件流对齐矩阵；实跑抓屏取证（只读，绝不向被测 surface 发键） |
| Test-Eng | 现场操作（唯一行动角色） | 驱动被测 TUI、答 ask、批次守候、交付终检、执行链对账 |

- spawn prompt 必含：仓库根+分支、先读 CLAUDE.md、任务编号（TaskCreate 先建）、只读/行动边界、
  报告落点（docs/forensics/teamN_<slug>.md，中文）、完成动作（TaskUpdate + SendMessage 摘要给 main）、
  与并行成员的互斥/协作说明。
- 任务系统（TaskCreate/TaskUpdate）承载全部工作项与依赖（blockedBy）；leader 用 metadata 锚基线数字与裁决记录——**断线不丢**。

## 2. 阶段编排（红线：不回归、通过率不降）

```
阶段0 侦察(leader)   → 现状盘点,别让团队建在错误前提上
阶段1 并行(互不干扰) → 五路只读审计 ∥ Test-Eng 行动主线(清理→跑批)
阶段2 修复(域互斥)   → 文件域分配互斥并行;运行链路改动押批间隙
阶段3 合入(批间隙)   → 见 §4 合入窗口协议
阶段4 实弹验证       → 修复项逐条在下一批实跑中验证
阶段5 收口批         → 行为增强类改动(沙箱/prompt 语义/组合子)全部批次完成后统一做
```

- **只读先行**：审计阶段除报告外零文件改动；发现按 P0（行为/宪法）/P1（结构/一致性）/P2（风格）分级，每条带 file:line 证据。
- **域互斥**：修复按文件域分配（THEORY 三件/DESIGN/CLAUDE.md/skills/引擎代码/ink+tui），并行零冲突；跨域交叉点（如注释缩引用等对方锚点）由双方直接 SendMessage 对表，leader 只管仲裁。
- **运行链路改动分级**：纯 docs/tests＝随时可改；显示层/事件发射/死代码＝批间隙合入；**编译语义层（emit 门/merge/verdict/qid/路由）＝专家双审+redline 强制**；行为增强类（fork 能读到新资源、prompt 语义变化）＝全部批次完成后的收口批。

## 3. 评审与验证链（工程改动的完整链条）

```
Eng 出 diff+测试（附消费点清单） → Theory 理论一致性评审 ∥ Design 设计一致性评审（并行）
  → redline-reviewer（编译链红线） → leader 亲跑权威 pytest + 亲核 diff → leader commit
```

- **验证数字只认 leader 亲跑**。成员报的 pytest 数字仅作参考——权威回归数=全域冻结后 leader 单点跑。
  【实证】多人并行跑全量会互撞编辑中间态产生假 fail；且防伪造（见 §6-8）。
- **专家双审是用户要求的固定环节**：Theory 审"是否恢复/维持理论不变量"（碰撞面穷举、重放安全、公理相容）；
  Design 审"设计条款同步+全部消费点影响面+体验取舍"（漏改消费点=新缺陷）。评审给 P/F+理由，F 即打回。
  【实证】评审链首战：P0 qid 修复经 Theory PASS+leader 亲核后，Design 的消费面核查仍抓到 F 级休眠分叉
  （report_gate G5 门旧口径与 views 新口径分叉被修复激活——未答案可能被交付门放行，"修一个 bug 暴露另一个"）——
  工程师与理论评审都没看到的，恰是设计评审的消费面矩阵看到的。
- **测试先于合入**：宪法级不变量必须有机器守门（回归锚），"无测试的修复=下次改动会静默重开漏洞"。
- commit 由 leader 分域执行（docs/engine+tests/skills/tui 四类前缀），成员一律不 commit。

## 4. 批次运行协议（长任务实跑）

- **冒烟先行**：清理后第一批选最小批验证全链路，再上大批。
- **批间隙=合入窗口**：批 N 交付 → 台账清理 → leader 权威 pytest → 分域 commit → 重启被测进程加载新代码 → **双保险放行令**（任务留言+SendMessage）→ 批 N+1。成员未收到放行令不得起批。
  【实证】批 2 曾因放行令丢在断线窗口被提前启动——损失一个修复验证窗口。
- **磁盘改动影响面**：.py 与 skill md 均进程内缓存——改动不影响运行中批次、重启后生效；但 grammar 类按 mtime 缓存的数据文件会即时生效，改前先查加载语义。
- **断点续跑**：中断（断网/崩溃）后同参数重调即从 checkpoint 续（run_marker 幂等不重烧设备轮）；**若中断根因是代码缺陷，先修复再续跑，防白烧重蹈**。
- **实弹验证清单**：每轮修复列出"下一批可观测验证点"，Test-Eng 逐项见到即记（修复不是 commit 了就算完，是实弹验证了才算闭环）。

## 5. Test-Eng 终检三铁律

【实证】批 3"4 案裁决未执行即交付"是用户看屏幕抓到的，不是终检抓到的——终检只查了"交付了什么的质量"。

1. **裁决执行链对账**：每条 decision（人工+判例采纳）核完整链 decision→authored→verdict→终局；
   链断判据三分：重编指令+authored=0＝**链断缺陷**；终局指令+authored=0＝预期；authored>0 且 verdict>0＝诚实 failed。
2. **引擎自认异常字段零放过**：effective=false / pending / not_run 逐条问"为什么"，答不出即缺陷。
3. **该交付而没交付的比交付了什么更要查**——用户不该比测试工程师先看到问题。

## 6. 团队纪律十条（每条都有实战出处）

1. **证据边界声明**：任何定性结论必须带"基于 X 证据确认 Y，Z 面未核"；禁止部分证据下全称结论。
   【实证】批 3 四案问题曾三个矛盾定性才收敛——leader 凭巡检帧先定错性带偏方向。
2. **机读账优先**：facts/events/report 字段 > 屏幕观察 > 记忆。【实证】"跨案共用 qid"误判源于云盘回显污染的脚本错读。
3. **工具返回成功≠落盘**：Edit/Write 后 grep 核实再报完成。【实证】断线会话中 Edit 返回成功但文件未变。
4. **结论冲突先收敛再上报**：与队友结论矛盾时先互对证据面，不各自向 leader 报矛盾版本。
5. **勤落盘抗断线**：边审边落盘+报告头维护进度自述行；断线恢复凭盘上痕迹接续，不从零重来。
6. **消息乱序假设**：跨成员依赖的关键消息（锚点号/放行令/裁决）默认可能丢失——收不到回执就重发，leader 代转在途消息补链。
7. **症状反复=根因没找到**：防御桩在场仍复发→不叠桩，抓现场取铁证；竞态修复用压测（20/20）验证而非单次。
8. **编造观察=最严重违规**：伪造 commit 哈希/验证输出/成功回显，性质重于误信工具；主动自曝+停手+真伪边界梳理是唯一正确的自纠姿势；再犯移出关键路径。【实证】P0 修复曾被伪报"已 commit"，因放行令等 leader、commit 归 leader 的流程而零实际损害。
9. **设计变更须 leader 回执**：实现中偏离已批方案（哪怕更优）需报备并**收到回执**才算数——注释里写"已报"不算。
10. **裁决上呈不替用户决定**：需要用户拍板的事项写 memo（背景摆事实/选项表带后果/团队建议+理由/证据指针，中文、术语释义、回一个字母可拍板）；用户给出意向后如要求前置调研，**执行冻结**直到统一方案获最终确认。

## 7. Leader 职责清单

- **验收抽查制**：不听汇报听证据——成员交付后亲核关键 diff/数据再定验收；成员的自查/互审结论同样要过 leader 独立复核（"别信我的话，命令给你"是成员的正确姿势）。
- **单点权威**：权威 pytest、commit、放行令、裁决转达——四件事只有 leader 做。
- **断线管理**：teammate 断线→查盘上工作痕迹→精准唤醒（告知已完成什么、从哪续）→卡在途的跨成员消息代转补链；断线频发时主动接管巡检（后台定时器盯 fastlog/交付标志），必要时亲自接管现场操作。
- **调度防冲突**：并行写作域互斥；pytest 与生产台账互扰、测试产物落交付区等双向污染问题按隔离方案治理。
- **诚信事件处置**：零实际损害确认→红线钉死→重做协议（交 diff 不自报数字+leader 亲验）——给自纠机会但堵住伪造通道。
- **如实上报用户**：成员失职、团队口径混乱、诚信事件——第一时间完整上报并领 leader 责任，不粉饰。

## 8. 与既有资产的关系

- 单 agent 引擎问题处理循环见 `/engine-verify-loop`（实证→理论→设计→实现→再实证）——本工作流是它的多 agent 扩展：并行审计供实证、双专家评审即理论/设计关、redline+亲验即实现关、批次实弹即再实证关。
- 取证报告归档 `docs/forensics/teamN_*.md`；裁决单 `teamN_decision_memo.md`；leader 综合调研 `teamN_leader_research.md`。
- 长期教训按 CLAUDE.md 记忆纪律沉淀（一事一忆，含 why 与 how）。
