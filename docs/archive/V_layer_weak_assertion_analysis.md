## 核心判断
grade **不是过严**。逐条读 CUT 理由，都是对的——draft 把"配置存在"当成了"行为生效"，
而需求要测的是动态行为。这正是项目红线 `no-per-case-hardcoding`（rr/wrr 的正确测法）要的。

## V 层弱断言四模式（均可 prompt 通用化）

### 模式1：静态配置检查代替动态行为验证
- grade[4]："需求要测的动态行为（rr 轮询顺序）**被大量静态配置检查淹没了**"
- grade[1]：draft 写 `found "rr"` —— 只验 `show sdns host method` 输出里 method 字段=rr，
  **没验流量真的在多 pool 间轮转**。
- 本质：`found <配置值>` 验的是"配了什么"，不是"行为发生了"。

### 模式2：断言不绑定目标对象
- grade[9]："`found 50` 只验证输出中有数字 50，**不绑定到 p2**。即使 p2 权重修改失败，
  只要 p1=30 在输出里，`found 50` 也可能误过"
- grade[22]："`found 172.16.35.213` 仅验证响应 IP 来自 server213，**未验证该 IP 是否确实
  属于权重 30 的 pool1**"
- 本质：断言期望值没和"它该归属的对象/pool/步骤"锚定，宽松到无法证伪。

### 模式3：动态序列断言缺项（只验头不验尾）
- grade[4]："**没有第二次 dig 的验证（命中 p2）**，无法证明'按照原有顺序命中'；步骤16/27
  只检查字面 IP，没验证是否命中'第 N 个'pool"
- 本质：轮询/权重/会话保持是 N 步序列行为，draft 只断言了第 1 次命中，缺后续命中点，
  无法区分"真轮转"和"恰好第一次对了"。

### 模式4：V 段来源全 intent，无手册/先例溯源
- grade[28]："全部 7 条 V 段断言 `source.kind=intent`，**无一引用手册或先例**"
- 本质：期望值是 draft"自己认为应该这样"，没落到先例的断言模式或手册事实，
  违反 observe-then-assert 红线的反面（凭空 assert）。

## PASS 的对照（好断言长啥样）
- grade[P2]："V 段断言**语义真覆盖需求要测的行为**，来源基本可信"——逐请求 IP + Hit 计数，
  咬住动态行为。
- grade[20]：好断言被夸"A 类核心行为断言（逐请求 IP + Hit 计数）**咬住了需求要测的动态行为**"。
- 共性：动态行为类需求，PASS 的都用了 **逐次请求 → 断言不同命中值 + Hit 计数绑定到具体 pool**。

## 改进：draft prompt V 段指引（通用方法论，非逐 case）
把原来一句话"轮询/会话保持类按确定顺序逐次断言"扩成可执行的自检四问，对应上述四模式。
**不写具体命令/期望值**（那是 draft 现场语义判断），只给"动态行为该怎么构造断言序列"的范式。

## 不能 prompt 解决的（信息缺失，诚实标注）
- 个别 case footprint/手册没有 pool 成员 weight 语法细节（draft 只能 grep 手册兜底，合理）。
- 期望值的具体数值（命中哪个 IP）依赖测试床事实 + 先例，是逐 case 语义判断，prompt 给方法不给值。
