# IST-Core 平台多根文件安全沙箱机制

在自动化测试与分析场景下，智能体（IST-Core）需要对本地测试资产、技术规格书进行直读，并在工作区直接输出评审建议文件、生成新的测试代码。如果对大模型的读写范围不加物理限制，极易造成大模型恶意格式化磁盘、读取用户敏感凭证或覆盘业务源码的重大安全事故。

为了在“授予智能体足够的文件掌控度”与“零信任用户安全防护”之间达到绝佳平衡，IST-Core 研发并固化了**多根沙箱白名单与多道拦截防御闸门**（核心在 `file_tools.py` 模块中实现）。

---

## 一、 控制台沙箱运行架构与多根优先权

主控智能体无法感知宿主机的绝对根路径，其面对的属于一个被深度治理后的虚拟文件系统结构。通过 `_agent_roots()` 函数动态配置，系统的只读和读写资源分布在多个具有前置优先级的物理白名单目录（Roots）：

```
[多根虚拟沙箱]
   │
   ├── Root A (只读权限最高): knowledge/data/  <-- 只读 KMS 转义标准化 markdown
   ├── Root B (普通读写优先): workspace/       <-- 用户输入、公共输出、过程缓存
   ├── Root C (会话临时区间): IST_SESSION_DIR <-- 会话状态专享
   └── Root D (用户级持久区): IST_USER_DIR    <-- 关联项目基础变量
```

### 物理检索与解析优先序列
当大模型下发一个读取路径请求（如 `read_file(path="product/ospf_guide.md")`）时，沙箱检索器会按照其前置优先级**依次在 A ➔ B ➔ C ➔ D 根目录下拼接寻找真实文件**。一旦首个目录存在对应路径，立刻返回，避免模型猜测相对复杂的工作目录结构。

---

## 二、 只读判定：防逃逸与拦截的三道拦截闸门

在智能体对任何路径进行“读”、“检索”、“列路径”以及“文件内容定位”行为时，请求路径都会被扔进 `_resolve_inside_root` 验证网进行检查，拒绝不合规调用：

```
大模型读取 filePath ➔ 【第一闸：Traversal 检测】➔ 【第二闸：黑名单库筛除】➔ 【第三闸：多根白名单投影】➔ 成功获取物理只读 path
```

### 1. 第一闸门：防 Traversal（跨目录逃逸）检测
任何带有磁盘盘符、转义符号、`..` 相对回退的请求将一律被首层剔除、静默拦截并直接向模型报告“权限受限或路径不合法”，消除利用特殊符号逃逸到系统关键目录的底层缺陷。

### 2. 第二闸门：顶层黑名单库排除 (`_PLATFORM_DENIED_TOP_LEVEL`)
即使路径不存在 `..`，只要其指向的绝对位置或者投影相对位置处于以下机密顶层目录和敏感文件系统范围内，将被直接拒止：
- 核心代码库：`main/`、`scripts/`
- 系统测试测试套：`tests/`、`pytest.ini`、`pyproject.toml`
- 长期记忆持久区：`memory/`
- 系统运行时状态：`runtime/`、`.web_server.pid`
- 凭证数据库：`ssh_users.json`、`environment`、`.git/`

### 3. 第三闸门：多根白名单投影
所有最终访问路径，必须能够被成功分解重构为至少一个多根列表（`Root A, B, C, D`）子成员的物理绝对子集。未在这个子集内的请求均视为非受权寻址。

## 三、 写入与删除判定：极度缩窄的限制四重锁

允许智能体修改和写入文件是智能体辅助开发的核心功能，但同样伴随着覆写甚至篡改系统、毁灭代码资产的极端风险。为此，IST-Core 定义了极其严苛的写操作前置路径断言（在 `_resolve_writable_path` 中定义）：

```
模型提请 write_file/edit_file ➔ 【第一闸门：防 Traversal】➔ 【第二闸门：黑名单排除】➔ 【第三闸门：工作区 workspace/ 根定位】➔ 【第四闸门：强制 outputs/ 子目录白名单】➔ 执行物理安全写入
```

### 强制 outputs/ 四闸物理硬控闩
系统强制规定：**哪怕是处于 `workspace/` 根目录下，智能体也绝对没有通用写入控制权。所有写入、修改、删除和新建脚本行为只被单向投影到 `workspace/outputs/` 这个唯一的叶子子目录中。**
- **知识库数据防污染**：任何想要写入 `knowledge/data/` 或 `workspace/inputs/` 的行为都会直接判定触发四闸逻辑并遭到物理拦截断言抛出，保障系统核心只读知识的永久权威性与免疫性。

---

## 四、 平台检索防御前缀机制 (`_RG_DEFENSE_PREFIXES`)

当主控模型调用 `grep_search` 或文件大范围批量匹配等涉及底层使用高速检索核心（如 `ripgrep`）的工具时，安全机制还会在检索前置注入隔离过滤器，直接对以下特征路径进行自动前置静默排除：
- 排除 `knowledge/.intermediate/`（避免模型将转码暂存区混淆为实际权威文本）
- 排除 `.cache` 及系统的隐藏标记资产

通过这一系列多防线、低侵入性的拦截闭环，保证了 IST-Core 即使搭载通识理解能力超强、代码编排意图丰富的大参数巨无霸模型，其在开发环境中运行的系统安全边界也坚如磐石。
