## 一、总体质量判断（一句话）

**这批 excel 的质量强弱是沿「被测行为确不确定」这条轴分布的，与用例难度无关：**

- **确定性行为（配置增删查改 / show 展示 / GA 优先级 / 边界上限）—— 质量好**。断言干净：`abs_found`/`not_found`/`found` 打在配置存在性与结构上，没有脆弱计数，意图覆盖到位。交付卷 22 案里约 15 个属此类，全部真通过、且断言经得起看。
- **非确定性分布（RR / WRR 轮转）—— 质量弱点，且失败集中于此**。excel 在这里反复犯三类断言毛病：①对 WRR 用**精确命中计数** + 假设「前 N 个请求全命中最高权重池」（错分布模型）；②对 RR 用**写死成员 IP + 假设确定轮转顺序**（req1 必中 p1）；③把 `not_found→found` 的时序锚点铺在**与轮转周期自相矛盾**的请求序列上。
- **另有一类失败不是断言问题，是配置构造缺陷**：wrr 配置漏 `priority` 参数→整条配置被设备拒，断言根本没机会跑。

**结论评级**：确定性用例的 excel 达到「可交付、断言可信」水平；**分布类用例的 excel 是这批的真正短板**——即使这轮修法让编写孔认对了「分布是采样不是成员」，落到断言构造仍系统性偏脆或偏错。

按断言算子体检，两组泾渭分明：

| 组 | 代表案 | 断言形态 | 质量 |

## 三、未成功卷（10 案）逐案：excel 质量 vs 设备/环境

**把失败分成三性质：〔A〕断言本身有缺陷（excel 质量问题）／〔B〕断言合理但配置构造缺陷／〔C〕断言质量好、失败源于设备真实行为或环境。**

### 〔A〕断言本身有缺陷 —— 3 案（真·质量问题）

**593484 wrr 选池** — 意图：wrr 权重 3:2:1，客户端1发3次→命中权重3的池，客户端2发2次→命中权重2的池。
生成断言：3 次 dig 后 `show statistics pool p1` 断言 **`Hit:\s*3`**、2 次 dig 后 `pool p2` 断言 **`Hit:\s*2`**。
**问题**：这假设「前 3 个请求全部命中 p1、接着 2 个全命中 p2」——这是**错误的 WRR 模型**。WRR 3:2:1 是在一个 6 请求周期内按比例交织分配（如 p1,p2,p1,p3,p2,p1），不是「先把 p1 的 3 次配额用完」。精确 `Hit:3`/`Hit:2` 双恒脆。→ 这正是分布验证的经典错法。

**778012 新增 pool rr** — 意图：新增 pool 后按原顺序命中、新池最后才命中。
生成断言：8 次 dig，前 3 次 `not_found 172.16.35.225`（p4 成员）、第 4–8 次全部 `found 172.16.35.225`。
**问题**：3 原池 + p4 = 4 池，RR 周期=4，只有第 4、8 次命中 p4，**第 5、6、7 次应命中 p1/p2/p3、不该出现 225**。断言「4–8 次全 found 225」与 RR 轮转**自相矛盾**（第 5–7 次必假）。写死成员 IP + 错轮转模型。（叠加配置前置 host 未建。）

**777976 rr 选池** — 意图：客户端1发1次→命中第1池，客户端2发1次→命中第2池。
生成断言：dig routera→`abs_found 172.16.35.213`(p1 成员) + `Hit:\s+1`；dig routerb→`abs_found 172.16.35.231`(p2 成员) + `Hit:\s+1`。
**问题（中等）**：假设 RR **确定性顺序**（第 1 请求必中 p1 的具体 IP、第 2 必中 p2）+ **精确 Hit:1** + 跨两客户端共享全局计数。结构方向对，但把非确定轮转钉成确定序列，偏脆。设备回显 `Hit:0`（统计根本没涨）说明假设没成立。

### 〔B〕断言合理，配置构造缺陷 —— 2 案（配置问题，非断言）

**572672 show 展示域名算法** / **572708 删除域名算法** — 意图：多域名配不同算法后 show 查看 / no 删除。
生成断言：`show sdns host method` 后 `found`/`not_found` 打每个域名的算法行——**断言写法完全正确**（确定性配置展示、结构断言、删除前后对照）。
**问题**：配置步骤给 wrr 算法时**漏了 `priority` 参数**，设备回 `The priority must be an valid value when the SDNS host method is "wrr". Failed to execute`——**整条配置被拒**，断言没机会跑。缺陷在配置构造完整性（wrr 必带 priority），不在断言。

### 〔C〕断言质量好，失败源于设备/环境 —— 5 案（excel 不背锅）

**778072 删除 pool rr** — 断言用**区间 `Hit:\s+[1-9]\d*`**（非精确）+ IP 择一 `(213|224)` + 结构检查 host 绑定（p1/p3 `found`、p2 `not_found`）。**这是全批构造最好的分布案之一**。失败于设备 `show statistics` 未按预期涨计数（设备 stat 行为），非断言逻辑错。

**593516 新增 pool wrr** — 断言用**区间 Hit≥1 + 20 次大量发包**（修法后的正确方向），验 p4 参与轮转。**构造方向对**，但设备回显 p4 成员 IP `172.16.35.225` 始终没出现（p4 未真正进轮转，构造/绑定问题）。弱点是没验「权重 40 应主导」、只验了 Hit≥1，覆盖偏弱，但不是恒真。

**681749 ga 算法** — 断言：两客户端各 1 次 dig 都 `found 213`(p1)，`p1 Hit≥1`、`p2/p3 not_found Hit`。**GA 确定模型下这是对的**（恒选最高优先级）。失败于设备返回 `Service IP:172.16.35.225`（没恒选 p1）——**设备 GA 行为与规格不符**，是真实发现，非 excel 错。

**778041 修改 pool 成员 rr** — 断言合理（改成员后 RR 顺序不变，req1→p1、req2→p2）。失败于整卷复跑**结论翻转**（先通过后不通过）= 环境/床不稳。

**994957 10 池关联上限** — 断言：`abs_found p1`/`abs_found p10` 验 10 个已关联，第 11 个 `found (?i)failed` + `found "maximum of.*10.*SDNS pool"`。**边界测试设计优秀**（上限内存在性 + 越限错误提示）。失败于执行中途某步失败致后续断言落空（vacuous），执行/环境问题。

## 四、结论

1. **整体质量**：确定性行为（配置 CRUD、show、GA、上限边界）的 excel **质量好、可信、真覆盖意图**——这是大多数（交付 22 + 未成功里的 572672/572708/994957/681749 断言部分）。
2. **真正的短板是非确定性分布（RR/WRR）的断言构造**：10 个未成功里，纯断言缺陷 3 个（593484/778012/777976）全是分布类，病根是**用确定性手段（精确计数/写死 IP/假设轮转顺序）验非确定分布**。这与「分布是采样、不是成员」是同一根问题的下一层——分类认对了，但**落到「发多少包、断言区间还是精确、时序锚点能否成立」的构造仍未收敛**。
3. **两类非断言失败**要分清，别记到 excel 质量账上：〔B〕wrr 漏 priority 是**配置构造缺参**（断言是对的）；〔C〕5 个案断言质量好，失败源于**设备真实行为（GA 未恒选最高、pool stat 不涨）或环境（复跑翻转、vacuous）**——这些 excel 反而有测试价值（暴露了设备/环境问题）。
4. **可操作的改进方向**（对 worker.md）：
   - 分布类断言**禁精确计数、禁写死成员 IP 钉轮转顺序**；WRR 用大量发包 + 区间比例，RR 用「命中集合∈{存活池成员}」而非「第 N 次必中某池」；
   - 时序锚点（not_found→found）只在**能真正构造出「先不在、后在」**的场景用（如新增/删除后），且请求序列必须与轮转周期自洽；
   - 配置构造门补 **wrr 必带 priority** 的完整性检查（这是机械可查的，不该漏到上机才暴露）。

**一句话**：这批 excel 对「确定性的东西」测得干净可信；对「分布轮转」测得偏脆或偏错——短板精确、可定位，且不是假验证蒙混，是构造方法没到位。

## 五、设备真实回显验证（用昨晚实机输出复核上面的判断）

> 证据源：每个未成功案 `workspace/outputs/dongkl/unfinished/<autoid>/attr_evidence.json` 的 `device_context`（昨晚实机逐条回显 + Fail/Success 明细）。9 个案有实机回显（777976/778012/778041/778072/593484/593516/681749/572672/572708）；593545/105941 是 0 轮欠定未上机、994957 执行 vacuous 无干净证据；**yzg 唯一非成功案 655233 也是 0 轮欠定挂起、未上机**——所以设备失败输出全在 dongkl 这 9 个。

实机回显不但**印证**了上面的判断，还揭出两条**比断言写法更深**的设备实况：

### 发现①：RR/WRR「第2个客户端命中第2池」这个前提，被设备实测推翻

| 案 | excel/脑图预期 | 设备实际回显 |
|---|---|---|
| 777976 rr | routera→p1(213)、routerb→p2(231) | p1 `Hit:1`+213 命中✓；**p2 `Hit:0`、231 找不到✗** |
| 778041 rr | 同上 | 同上：p1✓、**p2 `Hit:0`** |
| 593484 wrr | 3 请求 p1 `Hit:3`、2 请求 p2 `Hit:2` | p1 只 `Hit:2`、**p2 `Hit:0`** |

三案同一实况：**两个触发客户端（routera/routerb）不共享一个全局轮转计数**——client2 的请求也落在 p1，没轮到 p2。这不只是「excel 假设了确定顺序」，而是**脑图作者的预期本身**（"客户端2→命中第二个pool"）与本 build 设备的 RR/WRR 跨客户端行为不符（设备像是按源分别轮转/都从 p1 起）。excel 忠实编码了脑图的（可能过时的）预期，设备把它证伪了——这类"失败"其实是**测试正确地暴露了设备-规格差**。

### 发现②：设备 `Hit` 统计计数器不可靠

- 681749(ga)：dig 明明返回了 213（p1 成员，Success 1），但 `show statistics pool p1` 却是 **`Hit:0`**（Fail 2）——**返回了成员却不计数**。
- 593484：p1 返回 213、`Hit:2`；多个 pool `Hit:0`。

→ 大量断言压在 `Hit:\s+N` 上，本身就撞这个设备计数器的不确定性。**验分布不该只依赖 pool 命中计数**（这条比"精确 vs 区间"更根本）。

### 发现③：修法方向（大量发包+区间）在 593516 上实测是「成」的

593516（新增 pool wrr，权重 40）设备回显：**p4 `Hit:10`、p1 `Hit:6`**——p4 的权重-40 主导性被真实验到了，`Hit:\s+[1-9]\d*` 区间断言 + 20 次发包全部 Success。**唯一挂的 1 条**是把成员 IP `172.16.35.225` 找去了 `show sdns host pool` 的输出——而那条命令只列**池名**不列成员 IP。所以 593516 不是分布逻辑错，是**一条断言用错了命令**（成员 IP 该读 `show statistics` 不该读 `show sdns host pool`）。→ 修法的"大量发包+区间"方向被实机背书，残留是个窄的、可定位的错命令 bug。

### 发现④：配置构造缺陷被设备报错逐字坐实

- 572708 / 572672：`The priority must be an vaild value when the SDNS host method is "wrr"`——wrr 缺 priority，配置被拒，autotest2/autotest3 的算法根本没配上，show 自然找不到。**断言是对的，配置漏参**。
- 778012：`The SDNS host "autotest.com" does not exist. Failed to execute`——p4 绑定时 host 前置没建，p4 从未进轮转，后续找 225 全 fail。
- 778072：结构断言（p1/p3 在绑定、p2 已删、IP∈{213,224}、区间 Hit）**几乎全过**，只 1 条"删 p2 后单次请求命中 p1"挂（那次请求命中了 p3 不是 p1）——又是残留的单次轮转落点假设。

### 最终结论（设备证据加权后）

1. **确定性用例**（配置 CRUD/show/删除/上限）——excel 与设备一致、质量可信，无翻案。
2. **分布轮转用例**是真短板，但设备证据把病根从"断言写法"下推了一层：**① 不是所有客户端共享全局轮转，脑图的"client2→pool2"前提在本设备不成立；② 设备 pool `Hit` 计数器不可靠，不能作为分布验证的唯一支点。** → 验分布应转向「命中集合 ∈ {存活池成员}」+「高权重池成员在大量样本里占比显著」，**弱化甚至不依赖 pool 计数**，且**不假设特定客户端命中特定池**。
3. **修法有效性被实机背书**：593516 的 p4 `Hit:10` 证明"大量发包+区间"能真验出权重主导；残留只是错命令（成员 IP 读错 show）与错前提（跨客户端），非方向错。
4. **配置缺参（wrr priority）、前置缺失（host 未建）**是机械可查的构造缺陷，设备逐字报错坐实——应在 emit 门前拦，不该漏到上机。
5. **有几个"失败"是测试的价值而非缺陷**：777976/778041/593484/681749 实际暴露了「本 build 设备 RR/WRR 跨客户端行为、GA 恒选性、Hit 计数」与脑图预期的差异——这些应转成**设备行为观察（footprint）/ 缺陷候选**，而不是一味重编 excel 去迁就一个可能过时的脑图预期。
