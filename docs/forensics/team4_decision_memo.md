## ① 终验断批「缝合」的合法化条款——理论要不要认这个省时做法

**背景**。「终验」是交付前的最后一道确认：把本批全部已通过的用例合成一卷，在设备上**从头到尾连跑一遍**。理论规定（公式 15）交付只认这种整卷连跑的通过——因为用例之间可能互相影响，单独跑通过不代表连着跑也通过。「断批」指终验跑到一半被打断（网络闪断、人为停止），后半截用例没有跑到。目前实现（2026-07-16 提交，zhaiyq 批续跑实证）对断批采取了「缝合」策略：只补跑没跑到和失败的用例，补跑通过后与断批前已通过的前半段**拼接**成交付确认，不整卷重跑——实测避免了一次 7.8 分钟以上的整卷重跑及其引发的首案挂死；没跑成的用例（broken）不会被拼进确认名单。问题在于：理论文档里没有「拼接在什么条件下等价于整卷连跑」的条款——**实现先行了，理论缺一句话**，本轮审计将其列为待对齐观察项。

| 选项 | 内容 | 后果 |
|---|---|---|
| **A. 采纳缝合，理论补等价条款**（草文见下） | 在公式 (15) 处增补「缝合语境等价条件」，实现照条款逐条机械自查 | 理论与实现对齐；缝合的合法性从「默认成立」变成「每次逐条检查」；断批不再整卷重跑，省设备时间（实测每次 ≥7.8 分钟）且避免重跑挂死 |
| **B. 不采纳，回退实现** | 维持理论原样（交付只认整卷连跑），实现撤掉缝合逻辑 | 理论最干净；代价是每次瞬断/中止都整卷重烧设备时间，本批实证过一次 7.8 分钟以上并引发首案挂死 |
| **C. 维持现状（实现缝合、理论不写）** | 什么都不改 | 理论与实现继续脱节，后续每轮审计都会重报这一条；更重要的是缝合的前提条件**无人显式检查**——若断批前后床被动过（他人改了设备配置），拼接出的「交付通过」可能是假的而无人察觉 |

**建议条款草文（供选项 A 用，可修改）**：

> （15）补注·缝合语境等价条件：断批后「前段已通过凭证 + 断点后子集补跑通过」的拼接可视同整卷
> 连跑交付确认，当且仅当同时满足——
> ⒈ **同床同版本且执行连续**：两段在同一设备床、同一软件版本上执行，日志时间链无缺口，床账
> （设备状态台账）显示两段之间无第三方改动；
> ⒉ **断点两侧用例互不影响**：前段用例各自的清理步完整（已过配对恢复门），其残留不影响后段；
> ⒊ **没跑成的不算数**：broken / 未跑到的用例不计入已确认名单（现实现已如此）。
> 任一条件不满足 → 回整卷连跑重验。

**建议倾向：A**。理由：三个条件对应的检查手段全部已在系统里（执行时间链校验、床账、清理配对门），补的只是理论声明与逐条自查的接线；实证收益明确；选项 C 是三者中唯一带**静默假交付风险**的。

**证据指针**：`docs/THEORY_k_state_machine.md` §6 公式(15)、§2「合取③的语境参数」（用例独立性假设及其边界）、§5.5（语境锚）；`docs/forensics/team3_perf_audit.md` K5 行（缝合的成本收益账）；git 提交「fix(engine): 终验幂等闸断批例外——broken 组成不吸收（K5 窄化子集,zhaiyq 续跑实证）」；`docs/forensics/team4_theory_audit.md` §7（本观察项的审计原文）。

## ② 单元 E（评测基建）的去向——建 / 缓立 / 取消，三选一

**背景**。「单元 E」是设计文档 §18.15 立项的一个建设单元：给编译引擎建**评测基建**——机械化 benchmark（可重复的评测集）、naive 基线（对照组）、消融（量化每个构件的贡献），对应本项目「改 prompt/机制前先有评测，改完跑数字对比」的 eval-first 纪律。它现在处于**悬空**状态：§18.15 顶部横幅宣称「本节诸单元已被 dongkl 定稿取代」，但定稿对单元 E 零字对应——既没建、也没说不建（本轮设计审计定为 P0 级悬空）。与之相关的理论线：判断「编出来的用例是否忠实覆盖了测试意图」这件事，机械手段只能覆盖一部分（理论 §9-C1），全量判定只能靠独立评审器——而历史上的 LLM 评审器 grade 因判别力仅 3 个百分点（942 对实测）被整条删除；为防重演，理论 §7 设了 **DS-2「意图保真集」准入考场**（7 对带标签种子在盘，标签与上机结果独立）：今后任何评审器候选必须先在这上面量出判别力、达标才准上岗。现状是**考场立着、至今零候选来考**，且 7 对种子在统计上量不出可靠判别力（样本太小）。工程侧现状与之呼应（Design 签认）：回归防护载体零散（若干固定用例、金标准回放、门回归），无统一基准，六项外部框架对标中这正是我方最薄一环——此为 A 的立项依据；单元 E 是独立工作项，建与不建都不阻断引擎主路。另注：本项与第④项（DS-1/2/3 数据集承接）同属评测基建、共用原料，建议合并阅读后一并拍板。

## ③ config_generator 管线退役（整套约 1200 行）——删 / 修 / 挂牌保留，三选一

**背景**。`config-automation` 是一个把配置示例里的示例 IP 替换成环境真实 IP 的 skill。它附带一整套 python 管线：主脚本 `config_generator.py`（198 行）+ 配套 `modules/`（sdns/slb 两模块）+ `utils/`（拓扑解析/IP 映射/模块管理）+ 自带冒烟脚本，**合计 9 个 py 约 1200 行**（Design 复核实测 1187 行；原「880 行」为部分口径，已按现物更正）。本轮 skill 审计+Design 复核实测三件事：①**整套管线全仓库零外部调用方**（主脚本与全部 5 个配套模块逐一 grep：skill 目录外零引用；目录内仅自带冒烟脚本调主脚本，且该脚本不被 pytest 收集——即无任何生产/测试路径消费它）；②该 skill 的工具白名单为 `[fs_read, fs_write, fs_grep, fs_ls]`，没有 `run_python`/`run_shell`——agent 在正常配置下**根本无法执行**这套脚本；③skill 文档原先的示例传的参数在真实接口上不存在（文档已按事实重写：这项能力现在由 LLM 直接读拓扑文件、逐行替换、验证命令完成，全程不需要这套脚本）。即：脚本能力已被现行做法完全覆盖，且从证据看它从未在现行架构下被执行过。

| 选项 | 内容 | 后果 |
|---|---|---|
| **A. 退役删除** | 删除整套管线（`config_generator.py`+`modules/`+`utils/`+`scripts/`，SKILL.md 保留），工具激活映射里对应行同步清理 | 少约 1200 行维护债与误导源（新人不会再以为它在被使用）；git 历史随时可恢复；风险几乎为零（零调用方+无执行路径双证据，配套模块同为孤岛） |
| **B. 修复启用** | 补 `run_python` 白名单+重写脚本接口，让 skill 真走脚本管线 | 换回一条确定性管线；但需要真实需求牵引——目前 LLM 直执行方式在实跑中未见不胜任的证据，先修复再找用场是倒置 |
| **C. 挂牌保留** | 文件不动，头部加 DEPRECATED 标注 | 零工作量；债和误导源仍在，审计每轮重报 |

**建议倾向：A**。理由：零消费方+无执行路径+能力已覆盖，三个事实各自独立成立（Design 逐一复核通过，含配套模块）；删除可逆（git 历史）；B 需要的"LLM 直执行不够用"的证据目前不存在，真出现时再从历史恢复重写更贴需求。

**证据指针**：`docs/forensics/team4_skill_audit.md` P1-2 与 8-bis 节（零调用方 grep 证据、影响面分析）；`main/ist_core/skills/config-automation/`（重写后的 SKILL.md 与待裁脚本现物）。

## ⑥ prompt「行数预算」欠账的处置——减法轮 / 改预算 / 换约束，三选一

**背景**。设计文档 §5.5 给编写孔的指令文件（prompt）定过「行数预算」：worker 指令 ≤60 行、归因器 ≤50 行——出发点是「机械门已经强制的事不必再在指令里念叨」（门在则不教），指令越短越不容易带偏模型。同节还立了配套纪律：超预算必须**显性登记**（记一笔账：超了多少、为什么、拿什么换）。本轮审计实测：worker 指令现 **206 行**（预算 60）、归因器 **167 行**（预算 50），而账本停在 99/86——7 月 14 日到 17 日之间五轮功能增补（写保存族对策、题面三元组、对象链回补等）都没回填账本，「显性登记」承诺实质失守。需要说明的是：**这些超编行数多数各有实证理由**（每条对应一次真实翻车的对策），问题不在「写多了」本身，而在**约束机制已经名存实亡**——预算既没拦住增长，账本也没跟上事实。审计已把账刷新为真实数字，处置方式需要您定。

| 选项 | 内容 | 后果 |
|---|---|---|
| **A. 减法轮（删回预算内）** | 按 §5.5 既有检验逐条过堂（每条规则必须是：文法事实 / 理论投影 / 现行体制下实测仍复发的失效模式，三者之一，否则删），删到 60/50 行内 | 指令回到「门在则不教」的设计初衷，省 token；工作量最大（要删约三分之二）；删过头有恢复机制兜底（删除清单留档，失效模式复发即按数据恢复），但短期可能付一两轮「删了又要加回」的学费 |
| **B. 改预算（承认现实调宽）** | 把预算改到如 120/100 行，账本重新起算 | 登记纪律可重新执行；风险是预算失去牵引力——60 行的预算没守住，120 行的凭什么守住（约束本身没变强） |
| **C. 换约束（废行数、改「每行有据」）** | 废除行数上限，改为「每条规则必须在理论映射表里有归属行」（哪条理论/哪次实证支持它），可做成机器检查：规则条数 == 归属行数，对不上不放行 | 约束从「多少行」变「每行有没有依据」——直击真正要防的东西（无据散文），且可机器强制；风险是行数没有上限，若归属审查不勤，指令仍可能持续膨胀 |

**建议倾向：C 为主，可叠加 B 做熔断**。理由：①行数只是代理指标，真正的危害是「没有依据的防御性散文」带偏模型——C 直接约束这个，且与既有的逐条检验同构，可机器断言；②A 的一次性删除量太大且现有 206 行多数有实证归属，硬删是为了凑数字；③若您在意 token 成本，可在 C 之上叠一个宽松上限（如 150 行）当熔断线，越线必须先做置换再增补。无论选哪项，账上既有的一笔置换（检索顺序细节移入工具自身说明，指令里删除）可以先执行。
