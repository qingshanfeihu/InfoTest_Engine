## 面 3：收口批池理论风险扫描 —— **方向①高风险（第二真相源）；②③相容，建议②或③**

**P2-11 背景（team4_tui_audit.md:322-333）**：footer 业务 counts 编写期冻结 produced=0。**根因非 bug**：V8 account-实分离（INV-7）——author 节点 fork 逐个只写内存 `results[aid]`，authored 事实 barrier 后批量 append（nodes.py:457）；编写期 emit_tick 用节点开始的 fs 快照 → counts 现算恒 0，barrier 后跳真值。卡片走 fork_end 事件（实时）、footer counts 走 facts 视图（滞后）=**两层不同数据源**（#27 的「卡片 6 完成 vs footer 0/53」矛盾即此）。

**核心判据——counts 是业务态还是渲染态**：(42) 报告保真「每个计数必须可从事实流重算，render 零自由度」+ INV-7「业务态以事实流为准」共同要求：**对外显示的业务计数（produced/passed/failed/欠定）必须是 facts 派生**；fork 完成进度是**渲染态**（可用 fork_end 事件）。两者正交，混淆即公理冲突。

| 方向 | 理论风险 | 理由 |
|---|---|---|
| **① tick 快照并入 fork_end 结算数** | **高（除非严格限定）** | 若把**业务 counts** 改读 fork_end 事件累积 → footer 有了**事实流之外的第二真相源**（fork_end 走 .events.jsonl 渲染事件，非 facts.jsonl 业务事实）。致命点：**fork 完成 ≠ authored 入账**——fork 干完但 emit 门拒/入账滞后时，fork_end 说"完成 N"、facts 视图 produced 仍是别的数，footer 会显示**无法从 facts 重算**的数，counts 语义从"业务态"偷换成"渲染态"。直接违背 (42)+INV-7。**仅当**严格限定为"独立进度指示、绝不冒充 produced/passed 业务数"才可接受——但那已退化成方向② |
| **② 进度条按 fork 完成数填充** | **低** | 进度条是**渲染态指示**，用 fork_end 完成数天经地义；只要与业务 counts **分栏**、不把"fork 完成数"报成"produced 数"，业务 counts 仍 facts 派生 → 与 INV-7 相容。本质是承认"fork 进度"与"业务 counts"是两个正交量、各用各的源，不强行让 counts 实时化 |
| **③ 加「编写期产出计数将在合并时结算」提示语** | **最低（最合公理）** | 零数据源改动，counts 仍 facts 派生；不掩盖账实分离滞后，而是**向用户诚实解释**它。P2-11 的 User 体感问题（两帧不动像卡死）根源是"不理解为何滞后"，③直接解此，且完全不碰事件溯源 |
