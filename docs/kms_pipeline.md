# InfoTest Engine KMS 与直出 Markdown 知识管线

在传统的 AI RAG 知识平台设计中，开发者习惯性地去引入向量数据库（如 Qdrant）、文本切片算法、嵌入式模型（Embedding）和召回重排机制。但在工程落地的实践中，我们发现**对高度相关的专有名词、逻辑严格的测试用例 Excel，传统 RAG 的切片算法会彻底破坏其内部语义和跨工作表关联，导致大漏判和语义崩坍。**

InfoTest Engine 打破陈规，坚决执行**“去 RAG、去 Qdrant、全量高保真 Markdown 直出”**的极简高性能科学指引，让拥有长文本（Long Context）的智能体（IST-Core）直接通过高级检索（Grep, LS, Range PyReader）直接“阅读”完整的全局技术文档。

---

## 一、 为什么弃用向量数据库 Qdrant

1. **表格结构彻底破坏**：测试用例（`xlsx` 格式）拥有极强的行列表格关联（第一行定义字段，多行用例交叉，表格存在层级合并）。分片 Embedding 会将一整行割裂，大模型完全丢失了用例字段的表头关联认知。
2. **缺乏逻辑宏观拼图**：测试基线、通信 RFC 需要看全局的测试约束章回结构。RAG 会零散召回第 5 章和第 12 章的毫无关联的片段，导致 IST-Core 大脑脑补出南辕北辙的错误推论。
3. **完全一致检索效率高**：在海量文件或超大规模用例检索中，`grep` / `ls` 的速度（< 10ms）与准确率对精确的专用系统命名来说，在效率上完爆概率相关的向量语义检索。

## 二、 KMS 三级分桶分类管线的落差降级体系

在 `main/kms_classifier.py` 内部，系统实现了一整套自动的分箱落袋分类策略，用来将用户放置在 `knowledge/data/orgin/` 里的繁杂杂乱的源文档动态地归档到以下四个类别分桶（Buckets）：
- **`product`**：产品说明、指南、RFC 等技术规范说明文档。
- **`test_case_list`**：xlsx 格式的或者是纯测试用例目录表格。
- **`test_strategy`**：测试用例评审指导要点与分析方法文书。
- **`unclassified`**：未分类或是无法判断的兜底暂存包。

为了实现毫秒级的响应并消除云端大模型的无效分类调用费额，分类器建立了一套防波提级的**三级降级自适应路径**：

```
源文件分析请求 ➔ 【1级：用户覆盖配置】 ➔ 【2级：文件元数据与指纹缓存】 ➔ 【3级：LLM 轻量化判定】 ➔ 输出分类结果并自动持久化
```

### 1. 第一级：用户改写规则覆盖机制 (`.classifier_overrides.json`)
读取放置在 `knowledge/.classifier_overrides.json` 的静态名单。如果用户已经手动定义了某文件名的桶类别，则系统直接取用该白名单绑定值，响应开销为零。

### 2. 第二级：指纹热缓存映射 (`.classifier_cache.json`)
系统提取文件的**“文件名 + 文件最晚修改时间（mtime） + 文件字节字节（size）”**组合生成的唯一哈希指纹，高速比对放置在 `.intermediate/` 下的缓存映射。如果该文件没有内容变更，直接命中原缓存结果，大幅跳过重复计算。

### 3. 第三级：LLM 轻量智能判定模型
若前两级均未命中，系统通过后台 `function_llm.py` 请求百炼 / DashScope 的轻量化分析端点（`IST_HAIKU_MODEL` 或 `qwen-plus`），通过一段极其精练的系统级提示词和大模型思考，在毫秒级内得出带有置信度的 `ClassifierResult` 判别结果并自动归档：
- category（类别：product, test_case_list 等）
- confidence（置信评分：0.0 - 1.0）
- reason（分类理由解析说明）

分类判定完毕后，结果回写入二级缓存。当执行 `/kms product update` 或 `/kms qa update` 指令时，系统将按照确定的分类结果对文件实施针对性的转码重排输出。

## 三、 MinerU 高能 Markdown 批量渲染管线 (`mineru_batch_export.py`)

对于分类为特大型 PDF、Word 文档、或是演示幻灯片（pptx），普通的 pdf-reader 处理后格式杂乱，大模型根本无法辨析公式和复杂的插图拓扑。

### 精准解析与异步轮询机制
- **MinerU API 深度整合**：管线读取根目录 `environment` 配置文件中声明的的 `MINERU_TOKEN`，启用国内前沿的 MinerU 精准解构 API (v4 规范版本)。
- **切片与轮询**：文档首先被拆分投送给 MinerU 处理器，程序在后台多并发异步高频轮询 MinerU 的物理编译状态。当解析任务完成，自动将回执打包成的 zip 单据拉取回本地。
- **ZIP 解包与 Markdown 持久化**：解压 zip 将 MinerU 提取的高清晰规范 Markdown (`full.md`)、提取的多维技术插图自动写入到物理只读位置：`knowledge/data/markdown/{product|qa}/{stem}.md` 目标位置，使得长难技术大文档能够高精度直落，不丢失每一个技术换行与逻辑块。

## 四、 openpyxl 无 LLM 全自动用例表解析器 (`xlsx_to_markdown.py`)

大模型在处理 Excel（特别是带有单元格合并、多行嵌套字段的测试用例文件）时有着天然的劣势。如果我们去使用 LLM 逐行把 Excel 转成 Markdown，解析开销是天文数字，且生成质量容易脑补跑偏。

### GFM 高保真完全离线转换
- **直接使用 openpyxl 解释器**：程序采用纯 Python 的 `openpyxl` 引擎，抛弃 AI 分析，实现完全离线、毫秒级、零成本和零幻觉的高效直接编译。
- **GFM 规格化网格合成**：
  1. 探测合并单元格，对合并范围的每一列按绝对布局进行合并值复制填充（使得每行数据在 Markdown 化后独立成行，无信息残缺风险）。
  2. 提取表格实际大小并剔除空白虚拟区。
  3. 保障输出具有完全一致的 Github-Flavored Markdown 格栅表格，将大型 xlsx 输入一键落盘到对应的 `qa/` 桶内，使得大模型在通过 `read_file` 读入用例时，用例格式和表头行对齐如一，识别错误率由此直降为零。
