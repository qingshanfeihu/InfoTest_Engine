# 首选:blocks 组合子(比手写五列表更省心)

`compile_emit(blocks=…)` 是首选通道——你只做语义决策(测什么行为、用什么命令观测、选哪种断言形态、期望什么),寄存器分配/捕获比较三步式/E-F-H 列由工具展开。悬空断言、字面反斜杠 n、未定义寄存器在组合子语言下**写不出来**。传原生数组,不要序列化成字符串。

五种组合子(按要验证的事选):

```
{"kind":"CONFIG", "cmds":["命令1","命令2",…], "desc":"…"}
    设备配置。每条命令一个数组元素(换行工具拼,不要自己拼 \n)。单条→cmd_config,多条→cmds_config。

{"kind":"OBSERVE_ASSERT", "host":"routera|APV_0|…", "cmd":"dig/show 命令", "desc":"…",
 "asserts":[{"op":"found|not_found|abs_found", "pattern":"要在回显里找的文本/正则", "desc":"…"}]}
    观测一次 + 对该回显断言 1..n 条。验「输出里有/没有某内容」用这个。
    op: found=正则匹配(DOTALL) / not_found=不匹配 / abs_found=字面匹配(自动 escape)。

{"kind":"CAPTURE_COMPARE", "host":"routera|…", "capture_cmd":"第一次观测命令",
 "cmd":"第二次观测命令(省略=同capture_cmd)", "relation":"same|differs", "desc":"…"}
    捕获比较:第一次存寄存器、第二次产 result、断言两次 same(相同/会话保持)或 differs(不同/切换)。
    寄存器工具自动分配,你不用管。验「两次观测的关系」用这个(rr 轮转、会话亲和性)。

{"kind":"OBSERVE_ONLY", "host":"…", "cmd":"…", "desc":"…"}
    只发流量不断言(rr 轮转填充、发压力流量)。

{"kind":"SLEEP", "seconds":N}
    等配置生效。
```

host 语义:`APV_0`=被测设备(第一台);`APV_1`=第二台被测设备(双机场景,框架原生支持);其余=网络事实源里的测试机主机名(dig 类触发)。CONFIG 也可带 host 指定给哪台下配置(默认 APV_0)。

**provenance 按组合子粒度一对一**(steps 数与 blocks 数相等,一个组合子一条 {layer:G/E/V, source:{kind,ref}},展开工具同步)。

**rr/wrr 累计命中计数期望** 调 `compile_expected_hits(algorithm, n_requests, n_pools, uninterrupted, ...)` 算,不要手算——它带设备回放实证的适用域判定(连续查询段内精确/被 show 分段轮转态漂移只能按段/wrr 配比与权重不符只给参与性)。它返回数字段,你接在真实计数字段前缀后组成断言(前缀从先例/手册核实)。

组合子表达不了的边角形态,才退回下面的 steps 五列表手写。

# case.xlsx 步骤函数说明（写 steps_json 前先读这份）

一个测试用例是一张表，由若干「步骤行」组成。你用 `compile_emit` 的 `steps_json` 提交这些行——每行是一个 dict，核心是五个格：`E`（对谁）、`F`（做什么）、`G`（数据）、`H`（可选，把这一步的输出存进变量）、`I`（可选，把变量注入这一步）。这份文件把每个格能填什么、各是什么意思讲清楚（机制描述均以框架源码 `lib/test_xlsx.py` / `lib/env.py` / `lib/ssh_server.py` / `lib/client_action.py` 为准，mirror 有拷贝）。读懂它，你就能为任何行为设计出结构正确的断言。

## 一行 = 一步。必填三个格：E / F / G

### E 格：这一步对谁做（操作对象）

| E 填 | 意思 |
|---|---|
| `APV_0` | 被测设备(第一台,IP 见 network_topology.json)——给它下发配置命令 |
| `APV_1` | 被测设备(第二台,IP 见 network_topology.json)——双机场景(主备/递归转发/HA 对端)给第二台下发配置。框架原生双设备槽,与 APV_0 同法(cmd_config/cmds_config);单设备场景不要引入它 |
| `test_env` | 测试环境对象——F 填主机名,在那台机器上跑一条 shell 命令(dig 查 DNS、curl 发 HTTP、起停后端服务等) |
| `routera` / `server213` / `server231` / `server232` | 主机直连槽(框架 devices 表的一等键,`test_xlsx.py:176`)——E 直接填主机名,拿到该主机的 ssh 会话对象;F 填它的方法(`cmd` / `execute`,见下)。客户端侧 execute 动作走这个形态(`test_env` 对象上没有 execute) |
| `check_point` | 断言——检查「上一个观测步的输出」里有没有期望内容（不对谁做，是一次判定） |
| `time` | 等待 |

框架 devices 表还有 `APV_2`(第三台被测设备槽)、`Seg0_tmp/Seg1_tmp/Seg2_tmp`(segment 租户槽)、`http_server_231`(server231 上 http 服务的包装对象,`conftest.py:616`)——存在但当前拓扑/用例域用不到,列出仅为完整性。

### F 格：具体做什么（方法）。随 E 不同：

**E = `APV_0` / `APV_1`（给设备下配置/调预置动作）**
- `cmd_config`：下发**一条**命令，G 填命令原文
- `cmds_config`：下发**多条**命令，G 里用换行（`\n`）把多条隔开
- `execute`：调用**设备侧预置动作**（注册表见「execute 预置动作」节），G 填 `<动作名>：<参数>`
- （框架还有 `cmd_enable`/`cmd`/`array_config` 等方法，enable 态/root 态操作用，常规配置用例用不到）

**E = `test_env`（测试环境里的某台机器上跑命令）**
- F 填**主机名**。合法值是 `Env` 类的方法闭集（`lib/env.py:26-98`）：`clientc` / `clientd` / `cliente` / `routera` / `routerb` / `server213` / `server231` / `server232` / `console`。框架按方法名精确分发（`getattr` 无默认值，`test_xlsx.py:291`）——F 写错/写不在闭集里的名字 → AttributeError **崩整份文件**（emit 门会拦；大小写错误 emit 自动归一为小写）。
- 顺带的框架事实：**E 写错**（不在 devices 表）是另一种失败——框架**静默跳过整步**（不执行、无日志异常），后续断言拿错缓冲跑。E/F 两类拼错 emit 门都拦。
- G 填那台机器上实际跑的命令（如 `dig @<服务IP> <域名> A +short`）。client*/router* 是触发机（客户端侧发请求）；server213/231/232 是后端服务器（真实业务落点——在它们上面执行命令即可改变后端状态）。
- 触发机不止一台(事实源为准);需要"不同源客户端"这个变量时,用第二台既有触发机实现。
- ⚠ test_env 步里**任何 `ip addr` / `ip route` 的 add/del 都被 emit 必崩门整步拒绝**——主机网络状态由框架管理(`ssh_server.cmd` 对 add 自动记账、下一 case 开头 delete 恢复,地址与路由是同一契约的两半;自行增删会让恢复失败,崩整卷或以 RTNETLINK 残留污染后续 case)。要多源就用拓扑既有的多台触发机,别给主机加临时 IP/路由。

**E = 主机直连槽（`routera` / `server213` / `server231` / `server232`）**
- `cmd`：在该主机上跑一条命令,G 填命令原文——效果同 `test_env` 形态,差别只在对象拿法。
- `execute`：调用**客户端侧预置动作**(注册表见「execute 预置动作」节),G 填 `<动作名>：<参数>`。
- ⚠ `ip addr` / `ip route` 增删的框架管理契约对直连槽同样成立(同一个 ssh 会话类、同一条记账路径),emit 门同拒。

**E = `check_point`（断言）**
- `found`：上一个观测步的输出里**能找到** G（G 当正则,`re.DOTALL` 跨行、**无 MULTILINE**——`^`/`$` 只锚字符串首尾,而输出开头永远是命令回显行,行首锚永不匹配:found 恒 fail、not_found 恒真。要锚数据行用 `\n` 前缀。`check_point.py:23`）→ 通过
- `not_found`：输出里**找不到** G → 通过
- `abs_found`：和 found 一样找，但把 G 当**纯字面文本**（里面的 `.` `*` 等就是它本身、不当正则元字符）
- ⚠ 被检查的「输出」= **命令回显 + 数据 + 提示符**（框架 send(cmd) 后 read_until(prompt)，`ssh_server.py:cmd`）。模式若在**命令原文**上就能匹配（查询目标 IP、域名、命令关键字都在回显里），found/abs_found 恒真、not_found 恒 fail——什么都没验证，emit 门拦。写成只有**数据行**才满足的形态；结尾 `$`/`\Z` 锚同废（窗口末尾是提示符，emit 门同拦）。
- `found_times`：**禁用**（emit 必崩门拒绝）。框架对 check_point 只传 2 个参数（期望、输出，`test_xlsx.py:293-304`），这个方法要第 3 个「次数」参数，调用即 TypeError、崩整卷。旧说法「次数写在 I 格」与源码不符——check_point 里的 I 格是变量名（见「变量机制」节），不是次数。

**E = `time`（等待）**
- `sleep`：G 填等待的秒数

### G 格：数据（命令原文 / 要查找的文本）

- 配置步、请求步：填命令**原文**。
- check_point 步：填要在上一个观测步输出里**查找**的期望文本/正则。
- ⚠ **G 永远是字面文本或正则字符串——不是变量名、不是表达式、不是数值**。你写一个变量名（比如 `pool_ip`），框架就真的去输出里找 "pool_ip" 这几个字符，永远找不到。要填具体的值（IP 写成 `\b1\.1\.1\.1\b`：点要转义、两头加词边界，免得 `1.1.1.1` 误配到 `1.1.1.10`）。唯一例外：I 格注入时 G 里可以放 `{}` 占位符（见「变量机制」节）。

## execute 预置动作（两个注册表，按 E 分）

`execute` 把 G 当「动作调用」解析：取 `：`（中文冒号）前的文本做**动作名**，整个 G 传给动作函数，函数自己用 `：` 后的部分当参数、用中文逗号 `，` 分段（`lib/dic_operation.py:57-80`；写成英文标点解析不出参数）。动作是框架预置的封装（等待状态就位、批量配置核对、运行时值抽取等），一步顶手写多步。

**注册表是数据，不复制进本文档——编写前现查源码**（mirror 在盘上，注册表就在文件开头的 `command_function_mapping` dict，每个动作函数体几行、参数语法一眼可读）：

- 设备侧（E = `APV_0`/`APV_1`，F = `execute`）：`knowledge/framework/mirror/lib/apv/apv_action.py`（同义词表 `apv_synonyms` 同目录）
- 客户端侧（E = 主机直连槽，F = `execute`）：`knowledge/framework/mirror/lib/client_action.py`（同义词表 `client_synonyms` 同目录）

四条机械事实：

- **不在注册表里的动作名 = 静默跳过**：框架 print「未找到相似函数」后返回 None，这一步**不执行也不 fail**（`dic_operation.py:80`）。执行日志里看不出异常，断言会拿着空输出跑——写 execute 前先到源码核对动作名，照注册表原文抄。
- 模糊匹配（SequenceMatcher ≥0.8）的比较集只含**同义词表里的键和别名**——没进同义词表的动作只有精确写法一条路。
- 动作函数**有 `return` 的才产生回显/可配 H 存变量**；没有 return 的动作后面直接跟 check_point 是悬空断言（emit 门按注册表源码判定并拦截）。
- 抽取类动作没抓到目标时可能**静默返回哨兵值**（如 '0'）而非报错——拿返回值做断言前读一眼该函数体的 else 分支。

## H 格：存一次的输出、下次拿来比（只在「验两次观测的关系」时才用）

有些行为要验的**不是「某个值等于多少」，而是「第二次观测跟第一次相同还是不同」**——比如：
- 会话保持 / 亲和性 = 两次请求解析到**同一个**结果
- 轮转 / 切换 = 两次请求解析到**不同**结果

这种「关系」用固定的 G 写不出来：你编译时根本不知道第一次会解析到哪个，写死任何一个具体值都会偶对偶错。这时用 H 格——它把「第一次的实际输出」存起来，让后面的 check_point 拿它当比较基准（**存什么由设备运行时决定，你不用预测**）。

### 怎么用——三步，中间那步缺一步必崩

1. **第一步**：做一次观测（如 dig），这一行加 `"H":"v1"` —— 把这一步**整段输出**存进名为 v1 的格子。
2. **第二步**：**再做一次同样的观测，这一行不加 H** —— 这一步才产生「当前要被检查的输出」。
3. **第三步**：check_point 加 `"H":"v1"`、G 留空 —— `found` ＝ 这次输出和 v1 存的**相同**；`not_found` ＝ **不同**。

### 第二步的作用

带 H 的步只负责「把输出存起来」，它本身不产生「被检查的输出」。check_point 检查的是「它紧前那个不带 H 的观测步」的输出。带 H 的步后面要是直接跟 check_point，被检查的输出就是空的 → 框架报 TypeError、**整张表后面的用例全部不跑**。这就是「带 H 的观测」和「check_point」之间夹一个不带 H 的同样观测的原因。

形态（占位示意，不是要照抄的值）：
```
{"E":"test_env","F":"<触发机>","G":"dig @<服务IP> <域名> A +short","H":"v1"}   ← 存第一次
{"E":"test_env","F":"<触发机>","G":"dig @<服务IP> <域名> A +short"}            ← 不带 H，产生被检查的输出
{"E":"check_point","F":"not_found","G":"","H":"v1"}                            ← 这次≠第一次（在轮转/已切换）
```

`dig` 配 `+short`：只返回结果、去掉时间戳等噪声，存进 v1 的值才干净、好比对。

## 变量机制：H 存、I 取（框架源码事实，`test_xlsx.py:293-336`）

上面三步式是变量机制在「验两次观测的关系」场景的安全用法。底层机制本身更通用：

**H 格 = 存**。任何非 check_point 步都可以带 H：这一步的**返回值**存进名为 H 的变量（`locals()[H] = func(G)`）。观测步存整段输出；`execute` 步存动作的返回值（如 dnsperf 抽出的那个数字、抽出的 Session-ID）。

**I 格 = 取**。两种位置、两种含义：

- **非 check_point 步**：G 里写 `{}` 占位符，I 填变量名——框架把变量值 format 进 `{}` 再执行（`parameters[0].format(变量值)`）。变量名支持 `obj.attr` 点号取属性。这就是「上一步抽出来的值用进下一条命令」的通道：先用 H 存，再用 I+`{}` 取。
- **check_point 步**：I 填变量名 = 被比较的「输出」换成这个变量（默认是紧前观测步的输出）；H 填变量名 = 期望值换成这个变量（三步式用的就是这个）。也就是说 check_point 可以比较**任意两个已存变量**。

一步里 H、I 可以同时出现（先注入执行、再存结果）。变量**只在本 case 内引用**——这是编写纪律，不是框架限制（框架的变量表整卷存活、case 间不清空，但 case 必须自包含：子集复测按 case 拆卷跑，跨 case 引用在子集卷上必断）。引用没存过的变量名会 NameError 崩整卷（emit 门校验「引用必先捕获」，寄存器一律先 H 后用）。

## 分布区间断言：算法类（rr/wrr）发 N 次验命中分布

负载均衡分布类算法（轮询 rr / 加权 wrr）要验的是「流量在多个后端间按比例摊开」。**单次**请求命中哪个后端是运行时落点（算不准、写死会偶对偶错），但**发 N 次后各后端的累计命中分布**是离线可推的统计区间——rr 每后端≈N/k、wrr 每后端≈N×权重比，且各后端命中之和守恒等于总请求数 N。

这种断言不能写恒真的 `Hit:\s+\d+`（任意数字都过、没验分布），也不该整体标 `<RUNTIME>` 弃权（分布是离线可推的），更不能写死单次命中 IP。框架 `check_point` 没有数值区间算子，手写区间正则（如 8-12 要写成 `(?:[8-9]|1[0-2])`）极易错。所以用 **`dist` 声明**，让框架确定性展开成区间正则 + 守恒自检。

### 怎么写

一个分布断言写成一个 `dist` 步（`E=check_point`、`F="dist"`），它**紧前**要有一个产出各后端命中数回显的观测步（设备侧统计/计数命令，确切命令查 footprint/先例）：

```json
{"E":"APV_0","F":"cmd_config","G":"<查各后端累计命中的统计命令>"}
{"E":"check_point","F":"dist","dist":{
   "total":30,
   "field":"Hit:\\s*",
   "buckets":[
     {"anchor":"<后端1标识>","expected":10,"tol":2},
     {"anchor":"<后端2标识>","expected":10,"tol":2},
     {"anchor":"<后端3标识>","expected":10,"tol":2}
   ]}}
```

- `total`：总请求数 N（发包工具发了多少次）。
- `buckets`：每个后端一个。`anchor`＝把命中数锚定到该后端的标识（成员 IP/名字，依统计回显格式定；IP 记得转义点 `\\.`）；`expected`＝该后端期望命中（rr 填 N/k、wrr 按权重比，整数）；`tol`＝容差（按算法抖动 + 先例定）。
- `field`：命中数字段前缀（如 `Hit:\\s*`），依统计回显格式定。默认按 `anchor[^\n]*field<区间>`（后端标识与命中数同行）生成；回显格式特殊时，桶可给 `pattern` 模板（含 `{range}` 占位）自定义整条正则。

框架把每个桶展开成一条 `found` 断言，G 形如 `<anchor>[^\n]*Hit:\s*(?<!\d)(?:[8-9]|1[0-2])(?!\d)`（区间正则自动生成、带数字边界防 `12` 误配 `120`）。

### 两道确定性门（不过直接打回，按提示改）

- **守恒**：各桶区间必须能容纳总请求数——`Σ下界 ≤ total ≤ Σ上界`，且 `Σexpected` 应≈total（中心对齐）。期望/容差配得 Σ对不上 total 会被打回。
- **反恒真**：每桶区间上界必须 `< total`——否则单桶宽到「把全部流量都算通过」、对「算法失效＝流量全压一个后端」不可证伪＝恒真。容差太大触此门。

### 不适用

ga（优先级故障切换）、一致性哈希、会话保持是**确定性映射**（命中不摊开、同输入必同落点），不要用 `dist`——它们用上一节的 H 格捕获比较验关系。

## 命中归属锚点：这次命中了哪个 pool，别靠"两次输出同/异"猜

上一节的 H 格是给"这个值算不准"的场景用的——第一次落到哪个由运行时定，只能存起来跟第二次比同异。但**"命中了哪个 pool"往往不属于这一类**：一个 pool 里配了哪些成员 IP，是这份配置的静态确定后果（`sdns pool service` 写死了），"这次输出是不是落在某个 pool 的成员集合里"能直接判出来，不需要猜。

这个区分很关键，两种场景会把它们搞混：

- **pool 内多成员**：一个 pool 常配好几个成员 IP（pool 内自己也 rr）。"两次 dig 输出不同"可能只是同一个 pool 内部轮到了另一个成员，不代表命中了不同 pool——存 H、比同异，答不了"这次命中的是不是这个 pool"这个问题。
- **新增 pool 场景**：新增一个 pool 后，某次 dig 冒出一个此前没见过的新 IP，不代表这次命中了新 pool——它可能只是原 pool 里那几个成员这次刚好轮到了没轮到过的一个。反过来，只用 H 捕获比"这次跟之前见过的值都不同"，同样分辨不出"这个新值到底是不是新 pool 的成员"。

两种场景的共同解法：把"某个 pool 配了哪些成员"这个**静态已知**的信息，直接拿来当锚——这次输出命中了这个集合里的任意一个，就是命中了这个 pool；没命中，就是没命中这个 pool。跟前面①层（配置可推导）同源：池里配了哪些成员本就是静态常量，只是这里把它用作"归属判定"而不是"存在性断言"。

### 怎么写

`found`/`not_found` 的 G 本就是任意正则（本文件开头「G 格」一节），"输出 ∈ 成员集合"直接写成 `found(成员1|成员2|...)`；但手写多成员的 alternation 正则脆弱（IPv4 点位转义、词边界防 `172.16.35.226` 误配 `172.16.35.2264`、IPv6 冒号写法多样），所以用 **`member` 声明**，让框架确定性展开：

```json
{"E":"test_env","F":"routera","G":"dig @172.16.34.70 www.testrr.com +short"}
{"E":"check_point","F":"member","member":{
   "ips":["<该pool的成员IP1>","<成员IP2>", "..."],
   "present": true
}}
```

- `ips`：该 pool 配置里的成员 IP 集合（查 init 里 `sdns pool service` 配的那些；**只列这次查询类型能看到的成员**——dig A 只能解出 v4 成员，pool 里的 v6 成员在 A 查询下不可见，见下面「地址族」）。
- `present`：这次输出该不该命中这个集合。`true` → 展开 `found`（命中了这个 pool）；`false` → 展开 `not_found`（没命中这个 pool）。

跟 `dist` 不同，`member` 是 1 步 → 1 条 check_point（不是展开成 N 条），provenance 按位置对应即可，不用管展开量。

### new_member_last（新增 pool 最后才命中）这样排布

这类 claim 验的是"新增的 pool，要等原 pool 都命中过一轮才轮到"——不是"新增 pool 有没有参与"这种较弱的声明。用 `member` 声明按时间顺序排成一段 not_found、后面接一段 found：

```json
{"E":"test_env","F":"routera","G":"dig ..."}
{"E":"check_point","F":"member","member":{"ips":["<新pool成员集>"],"present":false}}
… 重复到覆盖原 pool 一轮（次数由原 pool 数/设备命中规律定，查 footprint/先例，不写死）…
{"E":"test_env","F":"routera","G":"dig ..."}
{"E":"check_point","F":"member","member":{"ips":["<新pool成员集>"],"present":true}}
```

这样写的可证伪性：新 pool 提前命中 → 前段某个 `not_found` 会 fail；新 pool 一直不命中 → 末尾的 `found` 会 fail。两个方向的缺陷都抓得住，不再被原 pool 的其他成员这种噪声蒙混过去。

### 地址族：dig 查询类型决定成员集合可见范围

一个 pool 混了 v4/v6 成员时，`dig <域名> A` 只能解出 v4 成员——命中 v6 成员那次，A 查询看到的是空/无关内容，不会在输出里出现该 v6 IP。用 `member` 锚这类 pool 时，`ips` 只列这次查询类型实际可见的成员（如只列 v4），或者让 dig 同时查 A 和 AAAA 覆盖到 v6 成员，否则会把"这次查询类型本来就看不到"误判成"没命中"。

### 不适用

单成员 pool（没有"pool 内部还有归属歧义"这层问题）、或本来就该用关系断言的场景（会话保持验的是"两次同一个客户端落到同一个后端"，不是"这次落到了哪个 pool"）——这些用前一节的 H 格捕获比较，不用 `member`。

## 两条贯穿全表的规则

1. **每个 check_point 紧前都有一个「产生输出的观测步」（dig 或 show，且不带 H）。** 配置步（cmd_config/cmds_config）下发命令后框架不收回显、不产生可检查的输出——check_point 紧跟在配置步后面会取到空输出，中间先用 dig/show 产生一段输出。
2. **每个用例至少要有一个能通过的 check_point**，否则上机必判 fail。
