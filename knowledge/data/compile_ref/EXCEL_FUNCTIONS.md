# case.xlsx 步骤函数说明（写 steps_json 前先读这份）

一个测试用例是一张表，由若干「步骤行」组成。你用 `compile_emit` 的 `steps_json` 提交这些行——每行是一个 dict，核心是四个格：`E`（对谁）、`F`（做什么）、`G`（数据）、`H`（可选，存一次输出、之后拿来比）。这份文件把每个格能填什么、各是什么意思讲清楚。读懂它，你就能为任何行为设计出结构正确的断言。

---

## 一行 = 一步。必填三个格：E / F / G

### E 格：这一步对谁做（操作对象）

| E 填 | 意思 |
|---|---|
| `APV_0` | 被测设备——给它下发配置命令 |
| `test_env` | 测试机——从客户端侧发起请求（dig 查 DNS、curl 发 HTTP 等） |
| `check_point` | 断言——检查「上一个观测步的输出」里有没有期望内容（不对谁做，是一次判定） |
| `time` | 等待 |

### F 格：具体做什么（方法）。随 E 不同：

**E = `APV_0`（给设备下配置）**
- `cmd_config`：下发**一条**命令，G 填命令原文
- `cmds_config`：下发**多条**命令，G 里用换行（`\n`）把多条隔开

**E = `test_env`（测试机发请求）**
- F 填**测试机主机名**（小写，如 `routera` / `routerb` / `clientc`）。框架按这个名字精确找机器——写错或写成大写会找不到机器、这一步根本不执行、没有任何输出。
- G 填测试机上实际跑的命令（如 `dig @<服务IP> <域名> A +short`）

**E = `check_point`（断言）**
- `found`：上一个观测步的输出里**能找到** G（G 当正则，跨行匹配）→ 通过
- `not_found`：输出里**找不到** G → 通过
- `abs_found`：和 found 一样找，但把 G 当**纯字面文本**（里面的 `.` `*` 等就是它本身、不当正则元字符）
- `found_times`：G 在输出里出现的次数，要等于 I 格给的数字

**E = `time`（等待）**
- `sleep`：G 填等待的秒数

### G 格：数据（命令原文 / 要查找的文本）

- 配置步、请求步：填命令**原文**。
- check_point 步：填要在上一个观测步输出里**查找**的期望文本/正则。
- ⚠ **G 永远是字面文本或正则字符串——不是变量名、不是表达式、不是数值**。你写一个变量名（比如 `pool_ip`），框架就真的去输出里找 "pool_ip" 这几个字符，永远找不到。要填具体的值（IP 写成 `\b1\.1\.1\.1\b`：点要转义、两头加词边界，免得 `1.1.1.1` 误配到 `1.1.1.10`）。

---

## H 格：存一次的输出、下次拿来比（只在「验两次观测的关系」时才用）

有些行为要验的**不是「某个值等于多少」，而是「第二次观测跟第一次相同还是不同」**——比如：
- 会话保持 / 亲和性 = 两次请求解析到**同一个**结果
- 轮转 / 切换 = 两次请求解析到**不同**结果

这种「关系」用固定的 G 写不出来：你编译时根本不知道第一次会解析到哪个，写死任何一个具体值都会偶对偶错。这时用 H 格——它把「第一次的实际输出」存起来，让后面的 check_point 拿它当比较基准（**存什么由设备运行时决定，你不用预测**）。

### 怎么用——三步，中间那步缺一步必崩

1. **第一步**：做一次观测（如 dig），这一行加 `"H":"v1"` —— 把这一步**整段输出**存进名为 v1 的格子。
2. **第二步**：**再做一次同样的观测，这一行不加 H** —— 这一步才产生「当前要被检查的输出」。
3. **第三步**：check_point 加 `"H":"v1"`、G 留空 —— `found` ＝ 这次输出和 v1 存的**相同**；`not_found` ＝ **不同**。

### 第二步的作用

带 H 的步只负责「把输出存起来」，它本身不产生「被检查的输出」。check_point 检查的是「它紧前那个不带 H 的观测步」的输出。带 H 的步后面要是直接跟 check_point，被检查的输出就是空的 → 框架报 TypeError、**整张表后面的用例全部不跑**。这就是「带 H 的观测」和「check_point」之间夹一个不带 H 的同样观测的原因。

形态（占位示意，不是要照抄的值）：
```
{"E":"test_env","F":"<触发机>","G":"dig @<服务IP> <域名> A +short","H":"v1"}   ← 存第一次
{"E":"test_env","F":"<触发机>","G":"dig @<服务IP> <域名> A +short"}            ← 不带 H，产生被检查的输出
{"E":"check_point","F":"not_found","G":"","H":"v1"}                            ← 这次≠第一次（在轮转/已切换）
```

`dig` 配 `+short`：只返回结果、去掉时间戳等噪声，存进 v1 的值才干净、好比对。

---

## 两条贯穿全表的规则

1. **每个 check_point 紧前都有一个「产生输出的观测步」（dig 或 show，且不带 H）。** 配置步（cmd_config/cmds_config）下发命令后框架不收回显、不产生可检查的输出——check_point 紧跟在配置步后面会取到空输出，中间先用 dig/show 产生一段输出。
2. **每个用例至少要有一个能通过的 check_point**，否则上机必判 fail。
