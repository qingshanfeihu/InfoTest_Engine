# 证据纪律

评审最容易翻车的点不是"漏看缺口"，而是"凭印象写结论"。以下规则是底线。

## 1. 链接 / 文件名 / 搜索命中不是证据

**反例**：
> "BUG-121100 已修复（参考 backup/knowledge/defect_cleaned/BUG-121100.json）"

只看到了文件名，没读内容。正例：
> "BUG-121100 已修复，详见 zentao/ZT-121100.json L23 'fixed_details': '...'"

必须 `read_file` 读到具体内容并引用关键字段。

## 2. "未检索到"不等于"不存在"

**反例**：
> "项目不存在 webui bug 122830 的相关记录"

agent 没法证伪。正例：
> "本轮未检索到 webui bug 122830 的本地证据。已搜过：knowledge/data/defects/zentao/、knowledge/data/defects/bugzilla/、grep 'webui.*122830' knowledge/data/markdown/。如需进一步追查，建议人工核对禅道。"

写清楚"实际搜过什么 + 接下来怎么补查"。

## 3. 引用用例必须给行号

**反例**：
> "用例覆盖了 5 种 ircookie_mode"

读者无法核对。正例：
> "用例行 55-59 覆盖了 5 种 ircookie_mode（plainname/hexname/ip/enc_name/enc_ip）"

引用产品文档同理：
> "`cookie methods.md` L23-45 定义了 IC/RC/EC 三种算法"

## 4. 每条缺口建议必须独立成行

**反例**（合并 7 项缺口为一条）：
> "建议补充并发、超时、日志、权限、多 group、WebUI 双语、reload 测试"

读者无法逐项跟进。正例（每条独立）：
- "**P1** 补充并发：行 207-282 group 算法测试增加 100 连接稳定性场景"
- "**P1** 补充超时：行 49-122 CLI 测试增加命令执行 30s 超时 Negative"
- "**P2** 补充权限：行 49-122 增加非管理员用户执行命令的权限控制 Negative"

每条独立 = 可分配 = 可跟进 = 可验证。

## 5. 评审基线是思考维度，不是逐条打勾清单

老版评审有 5 份 baseline JSON 把"必选子场景"写死成清单（边界 / 异常 / 并发 / 超时 / 日志 / 权限 / 多实例 / reload / 双栈 / 语言 / 主题），LLM 被迫每项打勾。**这是反模式**——产生大量"为打勾而打勾"的低价值建议。

新版做法：
- 这些维度作为**思考提示**记在脑里
- 只在"用例确实缺这个维度 **且** 产品功能确实涉及"时才提
- 比如：cookie 加密功能不需要"主题切换测试"——别写

判断"产品功能是否涉及"的依据是 Step 2 读的设计文档和 PRD，不是基线规则字典。

## 6. 建议必须含具体预期断言

**反例**：
> "补充 AES 加密测试，预期配置成功"

"配置成功"不是有效断言。正例：
> "补充 AES 加密正确性验证：配置 enc_name + 密码 A → 访问后获取 cookie 值 → 用密码 A 解密验证是否等于 RS 名称的 ASCII 值"

预期断言必须是可验证的具体行为 / 输出 / 错误码 / 比对结果。

## 7. 不要编造工具调用产物

agent 偶尔会"幻觉"工具返回了某个内容。每次工具调用后**先读返回结果**再下结论。如果工具失败（如 `web_bug_search` 远端超时），**明确写失败原因**，不要假装拿到了数据。
