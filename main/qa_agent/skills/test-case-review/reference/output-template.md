# 输出模板

四段式评审报告。每段都有具体格式要求。

## 一、读取到的证据

子段 1：缺陷信息（如有）

```markdown
### 1. BUG-XXXXX 缺陷信息
- **标题**：...
- **需求**：...（一句话概括）
- **新增/修改 CLI**：
  - `cmd1 ...`
  - `cmd2 ...`
- **新增参数 / 枚举**：...
- **影响版本**：trunk / 10.4.x / ...
- **关联 bug**：webui bug XXXX / doc bug YYYY（**必须列出，后续追查**）
```

子段 2：用例文件结构

```markdown
### 2. 测试用例文件结构（共 X 行）
- **行 1-N**：History + Option Definition（模板部分）
- **行 N+1-M**：List Template — 实际用例
- **行 M+1-X**：List Sample — 是否与本次评审相关？
```

子段 3：产品文档要点

```markdown
### 3. 产品文档 `xxx.md`
- 该文档讲了 ...
- 关键约束：...
- 与用例的关联：...
```

## 二、基于证据的判断

子段 1：覆盖良好（表格）

```markdown
### ✅ 覆盖良好的方面

| 维度 | 行号 | 评价 |
|------|------|------|
| **CLI 增删改查** | 行 49-122 | 覆盖了 ... 全生命周期 |
| **N 种枚举模式** | 行 55-59 | ... 均有用例 |
| ... | ... | ... |
```

子段 2：存在的问题（每条独立段）

```markdown
### ⚠️ 存在的问题

#### 1. <问题标题>（严重度：严重 / 重要 / 中等 / 轻微）
<问题描述：缺口在哪里 + 为什么这是问题>

**建议**：
- 补具体用例：<用例描述>
- 预期断言：<具体输出 / 行为 / 错误码>
```

每条独立成段。**禁止合并成"还需补充并发/超时/日志/权限"这种笼统列举**。

## 三、证据缺口

```markdown
| 缺口 | 说明 |
|------|------|
| 本地 spec 文档 | 未在 `knowledge/data/markdown/product/` 中找到 ... 的正式 spec 文档 |
| ... | ... |
```

写"未检索到"，**不写**"项目不存在"。

## 四、建议修改汇总

```markdown
| 优先级 | 建议 |
|--------|------|
| **P0** | 删除行 X-Y 的无关内容 |
| **P0** | 补充 ... 验证用例（含具体预期断言：解密结果 == 原始 RS 名称）|
| **P1** | 补充 ... 测试 |
| **P2** | 补充 ... |
| **P3** | 精确化预期结果描述 |
```

每条建议必须：
- 可定位到具体章节
- 含具体预期断言（不是"配置成功"这种）
- 独立成行
